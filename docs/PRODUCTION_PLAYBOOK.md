# 知名游戏网页化：端到端制作手册

> **状态：历史通用流程，已被后续实战手册取代。** 结合 CANOPY 实战校正后的当前规范，见 [CANOPY 开发总复盘与下一款游戏制作手册](CANOPY_DEVELOPMENT_RETROSPECTIVE_AND_NEXT_GAME_PLAYBOOK.md)。

> 适用范围：把一款已有大量公开资料的知名游戏，重构为可在浏览器中游玩的非官方体验  
> 目标：建立从命题、研究、设计、原型、生产、验证到 GitHub/Toy 发布与飞书通知的完整证据链  
> 核心规则：每一阶段都必须产生可检查制品并通过门禁；文档、截图和视觉包装不能替代可玩证据

## 1. 总原则

### 1.1 先定义体验，再定义功能

“有背包、有生命条、有制作按钮”不是体验定义。体验必须说明：玩家面对什么压力、观察什么信息、作出什么取舍、如何执行、世界如何回应、失败如何教会玩家下一局做得更好。

### 1.2 先证明循环，再扩内容

只有灰盒循环在无美术状态下仍然让测试者愿意继续玩，才允许进入纵向切片。美术不能挽救没有选择重量的规则。

### 1.3 原作忠实度是结构忠实，不是素材抄写

网页化应保留原作可识别的压力结构、信息结构和因果关系，同时根据浏览器输入、会话时长和性能预算重构表达。不得直接搬运受版权保护的原始素材、代码、音频或长文本。

### 1.4 每个判断都要可追溯

所有关键机制必须能追溯到：

```text
资料证据
→ 体验支柱
→ 设计规则
→ 可玩实现
→ 测试指标
→ 发布验收
```

### 1.5 “完成”只由门禁定义

Agent 报告“做完了”、页面能打开、截图好看、代码能编译，都不构成完成。只有阶段门禁的证据全部成立，任务才可进入下一阶段。

## 2. 制作体系总览

```text
S0 命题与授权
  ↓
S1 资料采集与机制拆解
  ↓
S2 体验支柱、范围与风险
  ↓
S3 纸面原型与数值模型
  ↓
S4 可玩灰盒
  ↓
S5 垂直切片
  ↓
S6 内容生产与系统扩展
  ↓
S7 Alpha / Beta / QA / 平衡
  ↓
S8 GitHub / Toy 发布 / 飞书通知
  ↓
S9 线上观测与复盘
```

任一阶段失败时，回退到产生该问题的最早阶段；禁止用下游包装掩盖上游缺陷。

## 3. S0：命题、授权与成功定义

### 输入

- 用户给定的目标游戏与“网页版”命题。
- 目标平台、设备、公开范围、时间和资源约束。
- 用户明确授权的 GitHub、Toy 和飞书操作范围。

### 必做工作

- 写出一句话产品承诺：谁，在多长会话中，获得什么独特体验。
- 明确作品性质：非官方致敬、机制启发、教育研究或商业产品。
- 明确不可触碰边界：商标、原始素材、音乐、角色形象、剧情文本和数据抓取规则。
- 定义网页化约束：键鼠/触屏、首载、帧率、离线、存档、嵌入容器和静态托管。
- 定义验收指标，而不是只写功能列表。

### 输出

- `PRODUCT_BRIEF.md`：目标用户、体验承诺、平台、范围、不做什么。
- `SUCCESS_METRICS.md`：乐趣、完成率、性能、可靠性和发布指标。
- `IP_AND_SOURCES_POLICY.md`：允许使用的资料和素材边界。
- 初始风险清单与决策日志。

### 门禁

- 所有人能用同一句话说明产品为什么值得玩。
- “网页版”已被转换成明确输入、性能和会话约束。
- 用户未授权的外部写入、发布或通知不会被默认执行。
- 至少定义 5 个可量化成功指标和 5 项明确不做的内容。

### 失败信号

- 只有“做一个像某游戏的网站”。
- 用页面栏目替代玩法目标。
- 还未确认 IP 边界就开始下载或复刻原作素材。

## 4. S1：资料采集与机制拆解

### 输入

- 已批准的产品命题和资料政策。
- 原作名称、版本、平台和目标模式。

### 资料分层

| 层级 | 资料类型 | 用途 |
|---|---|---|
| A | 官方网站、官方 wiki、开发者日志、补丁说明、商店页 | 确认版本、正式机制和开发意图 |
| B | 高质量 wiki、系统攻略、完整实机视频、速通资料 | 还原数值、步骤、边界和高级策略 |
| C | 论坛、社区讨论、玩家评测、失败案例 | 找痛点、隐性规则、元策略和体验语言 |
| D | 图片搜索、氛围参考、同类产品 | 仅用于视觉与交互参考，不作为机制事实单一来源 |

涉及技术事实时优先使用第一方资料；任何关键机制至少由两个独立来源交叉验证，或由完整实机录像直接观察验证。

### 必做工作

- 建立来源台账：URL、标题、发布日期、原作版本、资料类型、可信度、可使用范围。
- 将原作拆成六层：
  1. 玩家动词；
  2. 资源和物品流；
  3. 身体/战斗/环境规则；
  4. 空间与关卡；
  5. 目标、剧情与成长；
  6. 反馈、美术和声音。
- 记录典型 5 分钟、30 分钟和 2 小时体验序列。
- 记录失败原因、恢复路径、高手策略和新手误区。
- 区分“原作事实”“社区共识”“团队推断”“网页化改编决定”。

### 输出

- `RESEARCH/SOURCES.csv`：来源台账。
- `RESEARCH/MECHANICS_MATRIX.md`：机制、输入、输出、条件、反馈、相互作用。
- `RESEARCH/REFERENCE_FLOWS.md`：典型游玩流程和关键时刻。
- `RESEARCH/PARITY_MATRIX.md`：保留、改编、省略、待验证。
- 截图/视频时间码索引；不得将受限素材直接纳入发布包。

### 门禁

- 至少 100 条可追溯事实；核心支柱相关事实全部有来源。
- 至少完成 3 条完整实机流程拆解，而不是只看宣传片和百科摘要。
- 每个拟实现机制都标明来源版本和网页化处理方式。
- 团队能明确说出原作“去掉就不再是它”的三至五个结构特征。

### 失败信号

- 收集了大量截图，却没有机制矩阵。
- 只列功能名，不记录条件、代价、反馈和相互作用。
- 把社区猜测直接写成确定事实。
- Agent 用记忆填空，却没有引用或版本信息。

## 5. S2：体验支柱、范围与风险

### 输入

- 研究台账、机制矩阵、典型体验序列。
- 平台、周期、人员和技术约束。

### 必做工作

- 定义 3–5 个体验支柱，每个支柱必须包括：
  - 玩家感受；
  - 产生感受的规则；
  - 可观察的玩家行为；
  - 失败时的反例。
- 写出核心循环、短期循环、中期循环和整局循环。
- 建立“原作忠实度 × 网页可行性 × 乐趣贡献 × 成本”评分矩阵。
- 选择垂直切片：必须贯穿所有支柱，但内容量最小。
- 建立依赖图、风险清单和删减顺序。

### 输出

- `GAME_PILLARS.md`。
- `CORE_LOOP.md` 与状态/因果图。
- `SCOPE.md`：Must / Should / Could / Won't。
- `VERTICAL_SLICE_SPEC.md`。
- `RISK_REGISTER.md`。
- `TRACEABILITY.md`：证据 → 支柱 → 机制 → 测试。

### 门禁

- 每个支柱至少连接两个互相作用的机制，不能只是形容词。
- 核心循环中每一步都有玩家信息、输入、代价、反馈和状态变化。
- 纵切能在 15–30 分钟内展示全部支柱。
- 任一 Must 功能都能回答“删掉它会破坏哪个支柱”。
- 范围中明确列出不会实现的原作内容。

### 失败信号

- 支柱是“沉浸、真实、刺激”之类无法测试的词。
- 用页面区块、菜单和 HUD 代替核心循环。
- 首个里程碑就要求完整地图或大量美术。
- 所有功能都标为 Must。

## 6. S3：纸面原型与数值模型

### 输入

- 已批准的支柱、核心循环、纵切范围和成功指标。

### 必做工作

- 用卡片、表格或文本回合模拟玩家选择、资源变化和失败。
- 为每种资源定义来源、消耗、库存上限、替代路径和枯竭规则。
- 为每项动作定义前置条件、时间成本、风险、收益和反馈。
- 建立首版数值表，禁止将平衡常量散落在 UI 代码中。
- 使用种子 RNG 跑自动模拟，寻找无限收益、必胜、必死和无意义选项。
- 由未参与设计的人进行主持式纸面试玩。

### 输出

- `DESIGN/ACTIONS.csv`、`ITEMS.csv`、`RECIPES.csv`、`AILMENTS.csv`。
- `DESIGN/BALANCE_MODEL.xlsx` 或等价可计算模型。
- `DESIGN/PAPER_PROTOTYPE.md`。
- 至少 1000 局初步模拟报告。
- 纸面试玩记录和调整理由。

### 门禁

- 纸面状态下，测试者能感受到至少两次有代价的选择。
- 不存在已知无限净收益循环。
- 至少 3 条策略在目标胜率区间内可行。
- 资源流能解释玩家为什么探索、返回、制作和治疗。
- 所有数值位于数据层，可被模拟器与游戏共同读取。

### 失败信号

- 先写组件，再反推规则。
- 以“随机一点更好玩”替代概率设计。
- 只有奖励，没有机会成本。
- 纸面测试无聊，却决定靠美术和动画补救。

## 7. S4：可玩灰盒

### 输入

- 通过纸面门禁的规则、数据表和垂直切片范围。

### 目标

用最少视觉成本验证：移动与观察是否清楚、核心循环是否成立、压力是否可读、选择是否有重量。

### 必做工作

- 建立独立领域层：世界状态、实体、动作、事件、种子 RNG、目标和胜负。
- 建立最小空间：可移动、可返回、资源有位置，风险有预兆。
- 使用占位几何和基础音效，不制作宣传 Hero。
- 接入调试面板、事件日志、种子重放和遥测。
- 编写领域单元测试、模拟测试和一条关键 E2E。
- 每天进行内部试玩，每个可玩版本保留 build ID 与种子。

### 输出

- 可在浏览器直接玩的灰盒 URL 或构建包。
- 领域架构和内容 schema。
- 自动模拟、事件日志和性能基线。
- 至少两轮试玩记录及设计改动。

### 门禁

- 新玩家 3 分钟内完成一次完整核心循环。
- 不依赖解释文档，测试者能识别主要目标和一个主要风险。
- 10 名内部/邻近用户中至少 7 人愿意继续玩到 10 分钟。
- 输入反馈小于 100ms，关键结果反馈小于 300ms。
- 无 P0 规则错误；构建、类型、Lint、领域测试均通过。

### 失败信号

- 灰盒主要工作是字体、渐变、海报和营销文案。
- 世界仍由全局按钮代表，没有对象和空间上下文。
- 试玩者说无聊，团队回答“等美术做好就会好玩”。

## 8. S5：垂直切片

### 输入

- 已被证明可玩的灰盒。
- 美术、音频、UI 和性能预算。

### 必做工作

- 为一个完整 15–30 分钟段落制作接近发布质量的场景、UI、动画和声音。
- 打通从开局、教学、探索、采集、制作、威胁、恢复到结局/失败复盘。
- 定义视觉语言、素材规范、命名、压缩和加载策略。
- 加入无障碍、键鼠/触屏适配和低性能降级。
- 进行目标玩家盲测，不允许设计者现场解释。

### 输出

- 发布质量纵向切片。
- Art Bible、Audio Bible、UI 组件与反馈规范。
- 素材预算和性能报告。
- 盲测录像、数据与问题分级。

### 门禁

- 至少 20 名目标玩家盲测。
- 70% 以上给“好玩/愿意再玩”4–5 分。
- 60% 以上完成新手闭环，50% 以上主动开启第二局。
- 首局中位有效时长至少 15 分钟。
- 所有支柱在行为数据或访谈中都有证据。
- LCP <2.5 秒；目标桌面 60 FPS p95；无未处理异常。

### 失败信号

- 截图比游玩录像更有说服力。
- 只有一个高质量首屏，后续仍是占位按钮。
- 盲测失败却继续批量生产内容。

## 9. S6：内容生产与系统扩展

### 输入

- 纵切通过的规则、技术、美术、音频和数据 schema。

### 必做工作

- 只在成熟模板上批量生产地点、物品、配方、敌人、任务和事件。
- 内容数据与代码分离，建立 schema 校验和编辑预览。
- 为每项内容标注支柱贡献、出现条件、反馈、测试种子和预算。
- 建立素材清单、来源、许可、压缩状态和发布状态。
- 每周做全流程构建和内容审计，避免局部完成、整体不可玩。

### 输出

- 可版本化的内容包和素材清单。
- 内容覆盖矩阵、重复度报告、难度曲线和节奏图。
- 本地化表、无障碍文本和版权台账。
- 持续可玩的候选构建。

### 门禁

- 新内容复用已通过的系统，不引入平行假系统。
- 每项内容至少有一条自动校验和一个人工验收场景。
- 没有死配方、无用途物品、不可达目标或零反馈事件。
- 内容增加后，包体、首载、内存和帧率仍在预算内。
- 每周候选构建可从新存档完整通关。

### 失败信号

- 以物品数量、页面数量或代码行数衡量进展。
- 新增名词没有新的决策或相互作用。
- Agent 各自发明数据结构，导致内容无法组合。

## 10. S7：Alpha、Beta、QA 与平衡

### Alpha 门禁：功能完整

- 所有 Must 功能可达、可用、可存档、可恢复。
- 所有主流程有 E2E；领域规则分支覆盖率至少 80%。
- 无阻止通关的 P0；P1 有负责人、修复版本和回归用例。
- 种子化模拟可重复，至少跑 1000 局/候选平衡版本。

### Beta 门禁：内容与质量完整

- 内容锁定，只接受缺陷、平衡、性能、无障碍和发布修复。
- 至少覆盖 Chrome、Edge、Safari/移动端目标组合。
- 存档升级、损坏回退、离线/断网和 Toy 容器路径均验证。
- 完成首载、长局内存、后台恢复和弱设备性能测试。
- 完成版权、隐私、遥测和第三方依赖审计。

### QA 矩阵

| 类别 | 必测内容 |
|---|---|
| 规则 | 前置条件、资源结算、状态叠加、胜负、边界值 |
| 内容 | 可达性、重复、死物品、死配方、文本一致性 |
| 存档 | 新档、旧档、损坏、跨版本、云端/本地冲突 |
| 输入 | 键鼠、触屏、焦点、键盘导航、误触 |
| 性能 | 首载、FPS、内存、长局、资源释放、弱网 |
| 发布 | 子路径、静态资源、SDK、404、缓存、版本回滚 |

### 平衡门禁

- 任何单一动作占比不得长期高于 35%，除非它是明确的基础移动。
- 至少 3 条策略在目标难度下具有可接受胜率。
- 死亡原因分布不能由单一不可读随机事件主导。
- 新手失败应能从反馈中找到可执行改进，而不是只能怪随机数。
- 不存在永久无代价状态、无限净收益或可绕过所有威胁的安全策略。

## 11. S8：GitHub、Toy 发布与飞书通知

### 11.1 发布前输入

- 已通过 Beta 门禁的唯一候选 commit。
- 发布说明、已知问题、回滚方案、Toy 配置和通知对象。
- 用户对外部写入、发布和发送消息的明确授权。

### 11.2 GitHub 门禁

- 工作树范围清楚，不夹带用户无关改动或秘密。
- commit 信息说明产品变化和验证结果。
- CI 的 build、lint、typecheck、unit、simulation、E2E 全绿。
- PR/Release 关联 commit SHA、build ID、内容版本和验收记录。
- README 包含运行、测试、许可、非官方声明和已知限制。
- 发布包不含未授权原作素材、缓存、失败依赖目录或本地凭证。

### 11.3 Toy 门禁

- 使用生产构建，不上传本地开发目录。
- 验证相对资源、子路径、刷新、404、Toy SDK 加载和 CSP。
- 验证本地/云存档、登录与未登录回退、版本迁移和冲突选择。
- 生产 URL 返回成功，关键资源无 404，浏览器无未处理错误。
- 在桌面与目标移动视口完成一遍真实主流程。
- 记录线上 URL、发布时间、commit SHA、build ID 和回滚版本。

### 11.4 飞书通知门禁

只有以下事实全部被验证后才允许发“已发布”：

- GitHub commit/PR/Release 确实存在并可访问。
- Toy 生产 URL 已打开并完成 smoke test。
- 发布版本与候选 commit SHA 一致。
- 已知 P0/P1 为零，其他已知问题已写入发布说明。

通知模板：

```text
《项目名》版本 <version> 已发布

Toy：<production-url>
GitHub：<commit-or-release-url>
版本：<commit-sha> / <build-id>

已验证：首载、主循环、存档、移动端、无 console error
已知问题：<none-or-list>
回滚版本：<previous-build>
```

禁止在 URL 尚未验证、构建仍在进行或权限不足时发送成功通知。此时只能发送明确的“待发布/被阻塞”状态。

## 12. S9：线上观测与复盘

### 必须采集

- 首局开始率、教程完成率、首局中位时长、第二局启动率。
- 胜率、死亡原因、放弃位置、行动分布、配方使用率。
- 首载、FPS、未处理异常、存档错误和设备分布。
- 版本、种子和关键事件，以便复现问题。

### 发布后门禁

- 首小时完成 smoke 与错误检查。
- 24 小时内输出首份指标快照。
- 任何 P0 立即回滚或热修；不得用通知文案掩盖。
- 一周内做基于数据和玩家录像的复盘，进入下一轮 S2/S3，而不是直接堆内容。

## 13. 团队角色设计

一个长难游戏任务不能由“会写前端的单 Agent”包办。推荐角色如下：

| 角色 | 核心责任 | 不得自我批准的事项 |
|---|---|---|
| 执行制作人 / 根 Agent | 目标、范围、依赖、优先级、门禁和最终整合 | 不得用进度压力越过质量门禁 |
| 研究负责人 | 来源、版本、机制矩阵、原作体验拆解 | 不得直接决定网页改编方案 |
| 游戏/系统设计负责人 | 支柱、循环、规则、数值、胜负和模拟 | 不得用文档代替试玩 |
| 关卡/体验设计 | 空间、资源分布、节奏、教学和目标 | 不得只制作静态页面布局 |
| 技术负责人 | 架构、引擎、性能、存档、工具和发布约束 | 不得单独修改体验支柱 |
| 游戏客户端工程 | 世界、输入、实体、反馈与平台集成 | 不得在组件中私自发明规则常量 |
| UI/UX 负责人 | HUD、身体检查、背包、无障碍和跨设备交互 | 不得让 UI 文案承诺不存在的系统 |
| 美术/技术美术 | 视觉语言、素材管线、动画、预算和许可 | 不得在灰盒未通过时批量生产 |
| 音频负责人 | 环境、动作、威胁、状态和混音反馈 | 不得只做背景音乐而遗漏因果声音 |
| QA/平衡负责人 | 测试矩阵、种子复现、模拟、缺陷与玩家测试 | 不得由实现者独自关闭缺陷 |
| 发布/运维负责人 | CI、GitHub、Toy、监控、回滚和通知 | 不得在 smoke test 前宣布成功 |
| IP/安全审阅 | 版权、隐私、依赖、秘密和外部传输 | 不得因“非商业”默认放行素材 |

角色可以由 Agent 兼任，但责任和审批边界不能消失。关键设计、关键实现和验收至少由两个不同责任视角完成。

## 14. 多 Agent 协作协议

### 14.1 根 Agent 的职责

- 维护唯一目标、阶段、门禁和依赖图。
- 将工作拆成可独立验证的任务，而不是模糊地要求“完善游戏”。
- 在并行前声明文件所有权和接口契约。
- 汇总证据，不把子 Agent 的“完成”原样当结论。
- 发现上游假设错误时停止下游生产并回退。

### 14.2 每个任务必须使用任务契约

```text
任务：清晰、可验证的单一目标
阶段：S0–S9
输入：允许使用的文件、资料、接口、版本
输出：明确的文件、报告、URL 或测试证据
约束：不得修改的范围、IP、安全和性能限制
验收：命令、指标、截图/录像或人工测试门槛
依赖：前置任务和下游消费者
所有权：本任务可写文件与冲突规则
停止条件：何时必须回报阻塞而不是猜测
```

### 14.3 并行规则

- 只并行彼此独立且接口稳定的任务。
- 同一文件同一时间只有一个写入所有者。
- 研究、数值模拟、视觉参考和测试计划可以并行；核心状态模型未定时，不应并行实现多个相互依赖系统。
- 子 Agent 可以继续拆任务，但必须继承阶段、输入、约束和验收，不得扩张授权。
- 共享事实写入制品，不依赖聊天记忆。

### 14.4 交接协议

每个 Agent 的最终交接必须包含：

```text
做了什么
改了哪些文件/外部状态
用什么证据验证
哪些假设仍未验证
已知问题和风险
下游必须知道的接口/数据版本
建议的下一步，但不越权执行
```

没有命令结果、可玩构建、数据或可复现步骤的“已完成”视为未完成。

### 14.5 审阅协议

- 研究结论由设计负责人交叉审阅。
- 系统规则由 QA/平衡负责人用模拟和边界用例审阅。
- UI 文案由系统设计核对，禁止承诺不存在的机制。
- 关键代码由技术负责人和领域消费者双向审阅。
- 发布由非实现者执行 smoke test。
- 任何 Agent 都可以对门禁提出否决；否决必须附证据和回退建议。

### 14.6 决策日志

所有不可逆或高成本决定记录：

```text
日期 / 决策者
问题
候选方案
证据
决定与理由
影响的支柱、接口和制品
重新评估条件
```

“大家都默认如此”和“之前 Agent 这么做了”不是有效决策理由。

## 15. 反形式主义检查

形式主义指制品看起来完整，却没有降低真实产品风险。每次阶段评审都必须逐项质问：

### 15.1 文档检查

- 这份文档改变了哪个实现或测试决定？
- 每项关键结论是否有来源、数据或试玩证据？
- 删除这份文档后，团队会失去什么可执行信息？若答案是“没有”，停止维护。

### 15.2 功能检查

- 这是一个有条件、有代价、有反馈的玩家决策，还是只新增了一个按钮？
- 新名词是否与至少两个现有系统相互作用？
- 新物品是否有来源、用途、替代、消耗和失败路径？
- 玩家不看说明时能否感知这个系统存在？

### 15.3 视觉检查

- 截图之外，连续 10 分钟游玩是否仍然成立？
- 美术是否表达可交互性、危险和状态，还是只表达气氛？
- 宣传图和真实画面是否产生无法兑现的质量落差？
- 首屏是否把真正游戏推到折叠线以下？

### 15.4 Agent 检查

- 子 Agent 是在解决独立问题，还是为了展示“团队规模”而拆分？
- 输出能否被下游直接使用，还是只有一段漂亮总结？
- Agent 是否验证了运行结果，还是只根据源码推断？
- 是否因并行而重复研究、制造接口冲突或覆盖用户改动？

### 15.5 发布检查

- 当前 commit 能否从干净环境重建？
- 所谓“已发布”是否有真实 URL、SHA、build ID 和 smoke 证据？
- 飞书消息是在报告已验证事实，还是提前宣布愿望？

## 16. 一票否决红旗

出现任一红旗，阶段不得通过：

- 没有可玩构建，却声称游戏体验已完成。
- 核心循环只能用功能列表描述，无法用玩家行为描述。
- 纵切测试者需要开发者在旁解释才能继续。
- 资源不足按钮仍可执行，或 UI 文案与规则不一致。
- 存在无限净收益、不可治疗永久惩罚或无用途物品。
- 新增内容只增加数量，不增加决策或相互作用。
- 构建、Lint、测试任何一项红灯却准备发布。
- 使用未授权原作素材、秘密或个人信息。
- 生产 URL 未验证却发送“已发布”通知。
- 用海报、截图、长文档或 Agent 数量证明“好玩”。

## 17. 全项目 Definition of Done

只有同时满足以下条件，才可称“完成”：

- 体验支柱均有来源、机制、实现和玩家行为证据。
- 核心循环在灰盒阶段已经通过，纵切不是第一次验证乐趣。
- 目标内容全部可达、可理解、可完成、可失败、可复盘。
- 至少 20 名目标玩家盲测达到既定乐趣和重玩指标。
- 无 P0/P1；其余已知问题公开并有处置计划。
- Windows 和 CI 的 build、lint、typecheck、unit、simulation、E2E 全绿。
- 性能、无障碍、存档、Toy 容器和移动端达到预算。
- GitHub 制品、Toy 线上版本和 commit SHA 一致。
- 生产 URL 完成 smoke test，回滚方案有效。
- 飞书通知只陈述已验证事实。
- 复盘记录哪些假设成立、哪些失败，以及下一版本从哪个阶段重新进入。

## 18. 对本项目的立即应用

基于首版复盘，重制不能从“继续修改 `app/page.tsx`”开始。正确的第一批任务应是：

1. 冻结首版，只作为反例和视觉参考。
2. 完成有来源的原作机制矩阵与网页化 parity matrix。
3. 定义 3–5 个体验支柱和一个 15–30 分钟纵切范围。
4. 用数据表和种子模拟建立采集、制作、身体、天气和目标的纸面循环。
5. 新建独立领域层和真正有空间的灰盒，不沿用当前单组件状态模型。
6. 在灰盒通过玩家门禁前，不生产宣传 Hero、海报或批量内容。

这套流程的目的不是增加文档数量，而是让每一步都降低“最后做出一个不好玩的漂亮网页”的风险。
