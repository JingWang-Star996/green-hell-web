# CANOPY 开发总复盘与下一款游戏制作手册

> 项目：CANOPY: First Night / 雨林第一夜  
> 复盘日期：2026-07-16  
> 文档定位：把本项目从失败首版、系统重制、多轮玩家反馈到 Toy 候选发布的真实经验，转换为下一款游戏可直接执行的流程、门禁和模板。  
> 当前发布状态：同一候选包已经通过 B 站 Toy 审核，`mylist` 返回 `published`；正式地址已完成桌面首屏、既有存档恢复和游戏场景冒烟检查。

这份文档不是对 [旧版失败复盘](POSTMORTEM.md) 和 [早期制作流程](PRODUCTION_PLAYBOOK.md) 的重复。前者解释第一版为什么失败，后者给出通用流程；本文补齐后来真正开发、返工、测试、适配和发布中得到的经验，并以此作为后续游戏项目的默认手册。

## 1. 最终结论

我们真正学到的不是“怎样做一款《绿色地狱》网页版”，而是以下五件事：

1. **游戏不是系统名词的集合，而是一串玩家能读懂、执行、观察并学习的因果。** 有生命、理智、导演、生态、建造和存档这些名称，不等于对应玩法存在。
2. **先证明空间化核心循环，再扩内容、画面和发布。** 第一版像营销页，是因为先做了“看起来像游戏”的外壳；重制后才建立观察、准备、行动、后果、恢复的可玩闭环。
3. **世界必须只有一套真相。** 同样的树、石头、水、建筑和动物，外观、碰撞、焦点、动作、状态、掉落、刷新和存档必须指向同一个权威对象。
4. **玩家体验必须有独立证据。** 自动化测试能证明规则没有退化，构建能证明包能生成；它们不能证明首次玩家看得懂、手机上能操作、三小时不乏味或整体“好玩”。
5. **Agent 集群的价值来自清晰边界、证据门禁和统一集成，不来自角色数量。** 高耦合系统只能有一个最终写者；其他 Agent 更适合研究、审计、定向实现和独立复核。

对下一款游戏最重要的一句话是：

> 不要从“我们要做一款像什么的游戏”开始；要从“玩家在什么世界里读懂什么因果，用什么动作改变什么，并在失败后学会什么”开始。

## 2. 证据口径：以后禁止把“有代码”写成“已完成”

证据不是一条可以互相替代的等级轴。所有需求必须分别标记三个维度：

| 维度 | 标签 | 证明什么 | 仍不能证明什么 |
| --- | --- | --- | --- |
| 实现 | `I0 计划` / `I1 代码` / `I2 自动化` | 方向、实现、规则与回归 | 手感、可读性、部署 |
| UX | `U0 未测` / `U1 浏览器` / `U2 真机` / `U3 首次玩家` / `U4 长时` | 指定环境中的操作、理解、趣味和稳定性 | 未覆盖设备与人群 |
| 部署 | `D0 本地` / `D1 平台预览` / `D2 审核中` / `D3 生产` | 制品已到达哪个环境 | 该环境里的全部能力均已验收 |

例如 `I2/U1/D3` 只表示规则有自动化、完成过浏览器路径且制品已上线；它不表示真机、首次玩家或长时体验已经通过。

## 3. 项目演化：我们实际上走过了四次产品形态

### 阶段 A：像游戏的落地页

首版有标题、文案、状态条和按钮，但没有真实空间、目标对象、路线、工具、风险和持久世界。玩家能在很短时间内把按钮点完。它的问题不是美术简陋，而是产品形态错误：**它是一张描述游戏的页面，不是一款游戏。**

### 阶段 B：可通关的第一人称纵切（主线规则 I2、局部路径 U1）

重制后建立 Three.js 空间、确定性模拟、身体状态、资源、制作、营地、任务、死亡和结算。它证明了从处理伤口、安全饮水、建立营地、远征取电池到返营求救的规则路径，但没有证明首次玩家能自然理解或整体体验已经好玩。

此阶段又暴露了第二层问题：能通关不等于好玩。岩棚看不懂、模型交互点不够近、资源固定刷新、树石真假混杂、蛇是触发器、工具不具身、菜单和反馈失焦，这些都说明系统仍停留在“规则存在”，没有成为可信的世界体验。

### 阶段 C：Living Forest 基础（组件 I1/I2）

项目从线性任务 Demo 转向原创雨林生存：确定性区块、五类生态、资源再生、动物种群、显式装备、树木处理、岩石开采、自由放置、建筑空间语义、本地优先存档、恢复点时间线和移动端入口。这些是基础组件和局部纵切，不等于完整导演、完整生态链或三小时内容已经成立。

这个阶段最重要的变化不是功能变多，而是建立了三条目标架构原则；其中移动学/碰撞仍有渲染器权威例外，尚未完全收敛：

- 模拟决定主要世界事实，表现层只投影事实；当前移动和碰撞仍由渲染循环推进并定期同步给模拟。
- 每个世界对象有稳定身份，修改以差量保存。
- 第一人称物理动作使用可中断表现事务；模拟命令、休息恢复点和持久化各自使用独立事务协议。

### 阶段 D：发布候选与平台闭环（I2/U1/D3，生产深测待补）

项目补齐移动端功能入口、混合触控布局、保存导入导出、Toy 云配额、嵌套路径资源闭包、预览、审核提交和证据记录。这个阶段再次证明：**发布不是最后一条命令，而是一种独立产品环境。** 本地正常不代表宿主路径、SDK、权限、缓存、触控和云存档正常。

## 4. 第一版为什么不好玩：根因而不是症状

### 4.1 做了名词，没有做因果

首版有“伤口、寄生虫、理智、营火”等字段，却没有完整链条：

`信号 → 识别 → 选择 → 动作 → 世界变化 → 身体/资源后果 → 新知识`

只有一个字段增减时，它是仪表盘；当玩家能根据环境判断并改变未来时，它才是玩法系统。

### 4.2 没有空间，所以没有生存

生存玩法的乐趣来自地点、距离、视野、路线、负重、天气、工具、返程和营地之间的关系。把采集、休息、添柴和建造都放进全局菜单，会消灭这些关系。世界行为应该尽可能绑定具体对象和位置。

### 4.3 没有最小闭环，直接扩写了内容

正确的第一个目标不是“做完一堆系统”，而是让新玩家在三分钟内完成一次：

`观察对象 → 判断用途/风险 → 使用工具行动 → 看到世界改变 → 承担结果`

在这条闭环没有成立前，增加物品、植物、动物、任务和地图，只会复制更多不可信对象。

### 4.4 视觉对象与玩法对象双轨

早期 3D 世界一度存在“装饰树”和“可砍树”、“装饰石”和“可拾/可挖石”。相同外观对应不同规则会破坏玩家学习：玩家不再观察世界，只能依赖准星提示和隐藏例外。

修复原则是：

- 同形对象遵守相同基础动词；差异由工具等级、耗时、耐久、风险和产出表达。
- 不可交互填充物不能落入可交互对象的轮廓、尺度和高亮语言。
- 低画质可以减少装饰密度，不能移除或伪造玩法语义。

### 4.5 行动没有具身反馈

日志里写“已添柴”并不等于玩家添了柴。可靠反馈至少覆盖：

1. 输入被接受；
2. 工具或手开始行动；
3. 目标在接触点发生变化；
4. 声音、粒子、光照或模型同步响应；
5. 资源、耐久、时间和状态结算；
6. UI 解释结果或失败原因。

世界先反馈，UI 再解释。UI 不能代替世界变化。

### 4.6 数值没有经济关系

无限水、永久火、没有用途的物品、过快腐败、固定整批刷新都只是计数器。资源只有同时存在来源、用途、保存、损耗、替代方案和空间分布时，才形成经济。

### 4.7 任务与建造是伪自由

单实例建筑、固定点位和唯一顺序会把建造变成配方清单。自由建造的价值不在“可以拖动预览”，而在于建筑位置能改变当地的火、雨、休息、加工、储存、照明和路线。

### 4.8 包装与发布早于玩法证明

首版把首屏、文案和发布链做得比核心循环完整。正确顺序应是：

`体验支柱 → 因果蓝图 → 空间灰盒 → 可玩纵切 → 首次玩家验证 → 系统扩展 → 视觉升级 → 发布`

## 5. 已形成的设计原则

> 本节至第 10 节总结的是从 CANOPY 得到的**下一项目目标契约和默认方法**，不是 CANOPY 当前完成清单。具体完成边界以第 2 节和第 18 节为准。

### 5.1 三层循环套叠，而不是用材料数量拉时长

- **微循环（20–90 秒）**：看到信号，判断风险，选择动作，获得世界反馈。
- **中循环（8–25 分钟）**：营地诊断与配装，选择路线，远征，应变，返营加工与恢复。
- **大循环（45–120 分钟）**：建立前哨和安全走廊，解锁生态区与穿越能力，完成章节并改变世界阶段。

系统性新内容默认应改变至少两个系统。例如砍藤条不仅增加材料，还应制造声音、消耗耐久、改变植被状态或影响动物警戒。教学、无障碍、情绪节拍或纯粹提升操作清晰度的单一职责内容可以例外，但必须证明其不可替代的玩家价值。

### 5.2 任务检查世界事实，不检查脚本按钮

任务系统应记录“玩家已经取得安全水”“信标已通电”“某地标已调查”等事实，而不是要求玩家按固定顺序点击指定按钮。这样才能允许提前发现、不同路线、回头补做和熟练玩家加速。

每个任务阻塞必须回答四件事：

- 缺什么对象或状态；
- 需要什么工具或前置知识；
- 到哪里获取；
- 已完成多少、下一步是什么。

“结构不稳固”这类提示失败，是因为它没有说明真实几何条件、缺失支撑和修复动作。

### 5.3 危险必须可感知、可诊断、可反制

死亡可以严厉，不能黑箱。每种伤害都应拥有：

`环境前兆 → 直接警告 → 受伤来源 → 分级状态 → 处理方法 → 延迟后果 → 死亡因果`

蛇不能是半径触发器；它必须能被看见、听见、提前攻击、受伤、死亡或逃离。动物和陷阱同理。

### 5.4 导演不是许愿机

资源导演的职责是从**已经合法、符合生态、到达刷新时间且不在玩家视野内**的候选中排序和选择，不是把玩家当前缺少的物资直接刷到脚边。

可靠导演需要同时约束：

- 地理适配：河边有散石，湿地有纤维，林下有干叶；
- 生态承载：数量受群系、季节、天气和采集压力影响；
- 进度可达：关键治疗和初级材料在造成致命卡点前可找到；
- 视野公平：刷新离屏、离开区块或重新进入后物化；
- 稀缺诚信：任务物件、稀有资源和树木不能被导演瞬间补齐；
- 可复现：相同种子、时间和世界状态产生相同结果。

在 CANOPY 这类写实雨林中，树木适合以树桩、树苗、幼树、成树分阶段恢复；石头适合由河流冲刷、崩塌或节点暴露解释，而不是“长回来”。跨项目的普适要求是：再生规则必须符合本作世界语言、时间尺度和玩家预期。

### 5.5 建筑是本地基础设施

普通建筑应允许多实例，并拥有位置、朝向、碰撞、遮蔽、燃料、容量、维护和存档语义。建筑菜单负责选择蓝图，世界负责放置和使用。

一项建筑只有同时回答以下问题才算完成：

- 玩家为什么在这个群系需要它？
- 为什么建在这里，而不是任意位置？
- 它改变了哪两个以上的循环？
- 缺料如何显示和追踪？
- 建成后如何使用、维护、拆除和恢复？

### 5.6 时长来自新计划，不来自等待

三小时内容必须由新问题、新路线、新基础设施和新策略组成。禁止用大地图、重复采集、过度腐败、慢走和等待定时器伪造时长。

推荐节奏约束：

- 连续重复同一劳动不超过 12 分钟；
- 单一材料门槛不超过两个有意义往返；
- 每 8–25 分钟出现一次计划改变；
- 新章节必须引入新的环境问题或解决能力，而不是只换任务文本。

## 6. 世界、模拟与交互架构经验

### 6.1 模拟权威、表现投影是目标不变量

React、Three.js 和音频层不应各自判断游戏规则。推荐数据流：

```text
输入意图
  → 权威命令与预检
  → 固定步长模拟
  → 状态、事件与因果记录
  → UI / 3D / 音频共同投影
```

这使规则能用固定种子复现，也防止 UI 说可以、模型却碰不到，或模型显示燃烧、模拟却已经熄灭。

CANOPY 当前还没有完全达到这一目标：移动和世界碰撞由 `RainforestRenderer` 的动画帧路径处理，位置再按周期同步给模拟。下一项目应让模拟拥有运动碰撞，或定义明确的 renderer-kinematic authority contract，并对同步边界、碰撞重校验和存档位置负责。

### 6.2 一个可见对象，一套语义真相

每个离散对象应有稳定 ID，并共享：

- 生成来源与世界坐标；
- 外观家族和交互轮廓；
- 碰撞、遮挡和可达距离；
- 可执行动作与阻塞原因；
- 当前状态、生命/耐久/容量；
- 掉落、刷新与生态归属；
- 存档差量与恢复行为。

渲染层不能为了“更茂密”再生成一套看起来可采集但实际上只是装饰的对象。

模型尺度、可见操作面、交互锚、碰撞和使用距离必须由同一几何定义派生，并在极端尺寸、斜坡、第一人称视角和移动端视角下验证。气象柜、气象站和岩棚无法交互，正是这些参数分别硬编码后的结果。

### 6.3 动作需要分层事务，不能混成一笔跨系统大事务

第一人称砍树、开采、攻击和拆取适合使用表现事务：

```text
意图 → 预检 → 前摇 → 首次有效接触 → 模拟命令 → 恢复段 → 多通道反馈
```

同时还应区分：

- 模拟命令：一次不可变状态转移，制作和物品使用可直接进入这一层；
- 休息/关键流程 barrier：前后恢复点与 UI 收尾只在完整成功后提交；
- 持久化 checkpoint：按策略保存已结算状态，不与每一帧表现动作强耦合。

关键约束：

- 前摇期间条件变化可中断；
- 只有权威接触发生后才结算；
- 重复输入不能双重扣料或双重产出；
- 失败要保留明确原因；
- UI 只在成功提交或明确取消后关闭；
- checkpoint 不能落在“材料已扣但建筑未生成”的半状态。

### 6.4 大型确定性程序化世界默认保存差量

对于能由种子稳定重建的大型世界，推荐保存：

- `WorldManifest`：世界种子、生成器版本、时间、探索范围；
- `ChunkDelta`：采集、建筑、容器、尸体和生态变化；
- `KnowledgeState`：配方、物种、地标和永久事实；
- `QuestRun`：章节事实和计数；
- 玩家、营地、物品和恢复点。

基础地形与未修改对象由种子重建。稳定 ID 可采用：

`worldGeneratorVersion:objectFamily:chunkX:chunkZ:spawnIndex`

任何生成器升级都必须定义旧存档策略：兼容、迁移、固定旧版本或明确清档。不能悄悄改变旧世界的对象身份。小型世界、完全可变地形或生成器不稳定的项目，可选择完整快照、事件溯源或混合方案；核心是恢复后真值一致，而不是机械套用差量。

### 6.5 流式区块必须有预算和语义等价

- 渲染只保留邻近区块，模拟只运行活动区块；
- 远处动物保存为种群，近处投影为个体；
- 区块卸载必须释放 GPU 资源，但不能丢失权威状态；
- 低画质可以减少微型填充、阴影和粒子，不能改变可交互对象集合；
- LOD、实例化和对象池是性能策略，不能成为第二套玩法规则。

### 6.6 动态碰撞必须保留逃生路径

砍倒的树、生成的建筑和大型掉落会改变导航。每次动态碰撞变化都要验证：

- 玩家当前胶囊不被新碰撞包住；
- 有最小安全推出方向；
- 动画和权威碰撞切换在同一时刻；
- 读档恢复不会把玩家放进树桩、建筑或岩石内部。

本项目“砍树后卡进树桩”说明碰撞测试不能只检查生成物位置，还要检查**变化瞬间的玩家逃生空间**。

### 6.7 下一项目首日必须搭好的技术底座

- 持久化兼容元数据：`schemaVersion + contentVersion`；
- 世界身份：`WorldIdentity = (worldGeneratorVersion, seed)`；
- 周目身份：`RunIdentity = (runEpoch, worldGeneratorVersion, seed)`；
- 统一 `WorldQuery`：高度、河流、生态、碰撞、放置、视线和地图只能从同一世界场查询；
- 生成器版本路由：旧档缺版本按明确旧版读取，未知版本拒绝加载，升级通过迁移或保留旧生成器；
- 确定性场景运行器：记录种子、命令、帧号与状态哈希，一键复现玩家路径；
- 存档检查器：查看主档、备份、恢复点、云分片、校验和与迁移结果；
- 世界调试视图：显示稳定 ID、碰撞体、交互距离、导演评分和生成排除区；
- 浏览器 WebGL E2E：真实走通采集、制作、休息、死亡、读档、触控和 Pointer Lock；
- 性能采集器：记录帧时间分位数、Long Task、堆内存、draw call、三角面和区块同步耗时。

本项目仍有地形、河流、生态和移动同步边界没有完全收敛为一套世界真相；`GameClient`、`RainforestRenderer` 和主模拟文件也已成为耦合热点。下一项目应在第一次纵切前拆出稳定接口，不能等多人并行时再拆。

## 7. UI、输入与多端经验

### 7.1 多端要求动词等价，不要求排版相同

手机可以使用折叠抽屉、上下文快捷动作和更少的常驻信息，但不能缺少制作、装备、治疗、建造、存档和设置等游戏动词。

每个功能都应维护一张多端矩阵：

| 游戏动词 | 键鼠入口 | 触控入口 | 手柄入口 | 打开方式 | 同键/返回关闭 | 被 modal 遮挡时行为 |
| --- | --- | --- | --- | --- | --- | --- |
| 背包 | `Tab` | 生存菜单 | 待定 | 切换 | 是 | 高优先级 modal 先消费返回 |
| 制作 | `C` | 生存菜单 | 待定 | 切换 | 是 | 保留制作结果提示 |
| 暂停/存档 | `Esc` | 暂停按钮 | 待定 | 切换 | 是 | 加载/确认框拥有焦点 |

### 7.2 输入必须经过语义路由

不要在多个组件分别监听快捷键。统一路由应决定：

- 当前最高优先级 modal；
- 同一个键是打开还是关闭；
- Pointer Lock 是否应释放或恢复；
- 建造预览、加载对话框和暂停菜单谁消费 `Esc`；
- 键鼠和触控是否调用同一个动作命令。

“能用快捷键打开却不能同键关闭”是输入状态分散的典型症状。

### 7.3 UI 层级按生存优先级组织

推荐优先级：

1. 致命状态、直接伤害与交互结果；
2. 当前上下文动作和阻塞原因；
3. 任务下一步；
4. 常规状态和资源；
5. 日志、说明和低优先级通知。

全屏制作或建造面板不能遮住操作结果；危险提示不能永久占据手机画面。移动端“留意”应使用紧凑状态、渐进展开和可恢复详情，而不是常驻大卡片。

### 7.4 缺料反馈要转化为下一步行动

建造和制作界面应显示：

- 已有数量为绿色；
- 缺失数量为红色；
- 点击缺失材料可查看来源群系、可辨识轮廓、所需工具和加工链；
- 若资源当前不可能获得，必须显示真实阻塞，而不是保持可点击。

### 7.5 UI 缩放与可访问性是系统能力

1080p 默认字号、UI 比例滑杆、最小触控尺寸、横竖屏安全区、颜色之外的状态编码、无声危险提示和夜间基础可见性，都应在 UI 基础设施阶段完成，而不是后期逐页补丁。

## 8. 存档系统：玩家信任功能，而不是序列化函数

### 8.1 正确的保存模型

推荐同时具备：

- 手动存档；
- 多个轮转自动档；
- 睡眠前和睡眠后恢复点；
- 任务、地标、关键建造和退出恢复点；
- 死亡后可选择时间线中的恢复点；
- 本地文件导出、导入、校验和版本提示；
- 本地优先、云端异步同步；
- 主档、备份、损坏隔离和清档隔离。

### 8.2 本地成功不能被云失败回滚

保存协议应为：

1. 构造不可变快照；
2. 校验快照并保留旧主档/临时槽；
3. 写入新主档，再读回并校验；
4. 提交备份、恢复点 manifest 与索引；
5. 向玩家报告“本地已保存”；
6. 异步同步云端；
7. 单独报告云端成功、失败或待重试。

IndexedDB 可以使用真实事务；`localStorage` 只能实现可恢复事务协议，不能笼统声称原子写入。

云端超时、限额或 SDK 失败不能破坏已经成功的本地档。

### 8.3 冲突比较不能只看一个时间戳

至少使用：

- `runEpoch`：区分新周目，防止旧云档复活；
- `revision` / `tick`：比较同周目进度；
- schema 与生成器版本：决定兼容或迁移；
- checksum：识别损坏；
- source 与保存原因：供玩家理解恢复点。

标题页必须等待**有界的**云发现完成，避免旧云档在玩家开始新局后异步覆盖；同时不能无限等待宿主。

### 8.4 平台配额必须在设计时进入协议

Toy 云的键数、键名和值字节上限要求压缩、base64、严格 UTF-8 分块、manifest、校验和、旧块清理和失败关闭。平台限制不是发布脚本细节，而是存档协议的一部分。

### 8.5 存档验收必须跨越所有系统

每个持久状态都要经过：

`改变 → 离开区块 → 自动档 → 刷新页面 → 读档 → 导出 → 清档 → 导入 → 再读档`

至少覆盖树木阶段、矿体、植物刷新、动物伤亡、尸体、火、建筑位置与进度、容器、任务事实、知识、时间、装备和玩家位置。

## 9. 画面经验：学习 Valheim 的方法，不复制它的表面

Valheim 的价值不在“低多边形”三个字，而在于低成本资产被程序化世界、轮廓、地形层次、材质、光照、雾、天气、音景和构图共同放大。

本项目得到的顺序经验是：

1. 先统一世界语义，避免真假树石并存；
2. 再建立少量高辨识轮廓家族；
3. 用地形、水系、群系过渡和林冠层次塑造空间；
4. 用前中远景、巨大叶片、雾和声景制造密度；
5. 用实例化、对象池、LOD 和区块预算控制性能；
6. 最后做材质、顶点色、灯光、天气和镜头构图。

每项视觉升级应提升可读性、情绪、作品身份或空间记忆中的至少一项，并且不得损害交互辨识、性能和无障碍。若只是让装饰数量变多，却让可交互对象更难辨认，它是倒退。

当前项目已有程序化密度、区块、植物差异几何和视觉研究，但这不等于最终 Valheim 式画面升级已完成。

## 10. 测试经验：规则、表现和人类三条证据链

### 10.1 自动化应该优先验证不变量

高价值测试包括：

- 固定步长和固定种子的确定性；
- 所有生成的可交互对象都有可达动作；
- 同一对象的外观、焦点、碰撞和命令身份一致；
- 时间、腐败、耐久、刷新与休息共享同一时钟；
- 动作事务不会重复结算或落入半状态；
- 低画质与标准画质拥有相同可交互 ID 集；
- 存档往返、损坏隔离、迁移和云冲突；
- 动态碰撞变化后玩家仍可逃脱；
- 移动端和桌面端拥有同一游戏动词集合；
- Toy 嵌套路径下所有资源闭包可加载。

### 10.2 自动化不能替代首次玩家

规则测试可以通过，同时岩棚仍像一块山、干叶仍难找、夜晚仍太黑、菜单仍挡住反馈。每个里程碑都需要至少三类试玩：

- 开发者聚焦 smoke：刚修改的路径是否可用；
- 非实现者红队：寻找矛盾、隐藏规则和负面路径；
- 首次玩家盲测：不看设计文档，观察能否自己理解。

### 10.3 长时游戏必须做时长和恢复测试

三小时内容至少需要：

- 首次玩家墙钟样本；
- 熟练玩家快速路径；
- 长时固定种子 soak；
- 多次睡眠、死亡、读档和导入导出；
- 不同路线的资源可达性；
- 内存、GPU、区块数量和存档体积曲线。

### 10.4 不要把推荐门禁误写成当前 CI 已有能力

CANOPY 当前 CI 主要覆盖 typecheck、全量 Node 测试、lint、生产构建和 Pages 构建；Toy 构建/doctor、WebGL E2E、多视口触控、长时 soak、性能和 manifest—commit 绑定仍主要依赖发布时人工执行。下一项目应拆成三条流水线：

1. **PR 流水线**：跨平台静态检查、确定性测试、旧档兼容、生产构建、Toy 构建与资源闭包；
2. **浏览器流水线**：真实 WebGL 的桌面/竖屏/短横屏路径，覆盖触控、Pointer Lock、采集、制作、休息、死亡和读档；
3. **Nightly 流水线**：百万 tick、随机命令、长时内存/性能、存档语料和 manifest—commit 一致性。

## 11. 多 Agent 协作：怎样组织团队才不会制造更多返工

### 11.1 一个用户入口，一个最终集成者

主协调 Agent 负责：

- 理解和冻结玩家问题；
- 维护唯一需求矩阵、决策日志和发布状态；
- 拆分互不冲突的任务；
- 统一高耦合文件和状态机；
- 运行最终门禁；
- 对用户做准确、不夸大的交付说明。

不要让多个 Agent 同时修改模拟状态、核心类型、世界生成和主 UI 容器。Agent 数量增加不会自动增加吞吐量，高耦合冲突会迅速抵消收益。

### 11.2 按任务动态组队，不维持庞大常驻组织

一轮典型团队只需要：

- 研究/需求 Agent：把参考资料和反馈转换为可验收需求；
- 系统实现 Agent：在清晰文件边界内完成一个纵切；
- UX/存档红队 Agent：只读检查负面路径；
- 世界/玩法红队 Agent：检查可读性、因果和游戏性；
- 主协调 Agent：唯一集成与发布者。

任务结束即释放角色；下一轮根据风险重新组队。

### 11.3 每个子任务使用任务契约

```text
目标：玩家可观察到的具体结果
玩家假设：玩家应该理解、做到或感受到什么
基线：commit、构建、种子、存档和真值源
范围：必须完成与明确不做
写入所有权：允许修改路径、禁止触碰的耦合区域
非目标：本轮明确不做什么
输入：需求、复现、参考、接口版本
依赖：前置制品和下游消费者
输出：代码/文档/测试/截图或报告
验收：命令、步骤和可观察结果
证据标签：本轮最多能达到哪些 I/U/D 证据
外部权限：是否允许 GitHub、Toy、通知或其他外部写入
独立验证者：由谁复跑，是否必须非实现者
停止条件：完成、冲突、缺权限或发现范围扩张
交接：改动、证据、未验证假设、风险和版本
```

### 11.4 审计与实现分开

先让只读 Agent 找问题、建立严重度和复现，再安排实现。边查边改容易让审计标准随实现变化，也会让“已有代码”冒充“问题已关闭”。

### 11.5 独立复核不能由原实现者自证

至少对以下内容安排非实现者复核：

- 存档、导入导出和死亡恢复；
- 移动端入口与 modal；
- 世界对象语义和可达性；
- 资源导演和关键物资；
- 发布包、路径、平台状态与对外文案。

### 12.1 Toy 是嵌套静态宿主，不是本地根路径

首个 Toy 版本出现样式和脚本 404，根因是产物仍引用开发框架的根路径资源。发布包必须是位置无关闭包：

- 相对资源路径；
- 单一明确入口；
- 不依赖开发服务器路由；
- SDK 只加载一次；
- 嵌套 slug 下完成真实预览；
- 平台不支持的 feature policy 只能作为兼容警告，不能让游戏启动依赖它。

### 12.2 正确发布顺序

```text
冻结候选
  → typecheck / test / lint
  → 生产构建
  → Toy 专用构建与资源闭包检查
  → toy doctor
  → 生成预览（不提交审核）
  → 桌面、竖屏、短横屏、存档关键路径核验
  → 用户确认或使用预先授权
  → 提交审核
  → 记录真实 CLI 状态
  → 审核通过后生产 URL 与宿主能力复测
```

`auditing` 只能写成“已提交审核/审核中”，不能写成“已发布上线”。

### 12.3 GitHub、Toy 和通知必须对应同一候选

发布记录至少包含：

- commit SHA；
- 构建命令和结果；
- 包体文件数和字节数；
- Toy ID、slug、预览 URL、提交时间和 CLI 状态；
- 真机/浏览器矩阵；
- 已知未验证项；
- 通知内容和发送结果。

不能先通知“完成”，再发现推送的是另一份工作树或审核仍未通过。

本项目本地发布历史显示 2026-07-14 至 2026-07-16 对同一 Toy 至少执行了 6 次 `update`，其中 7 月 14 日一小时内 3 次。结合此前生产资源 404 和多轮宿主反馈，可以推断宿主适配和候选冻结介入过晚，让发布环节承担了部分本应在本地预览解决的 QA。下一项目必须在纵切阶段就维护 Toy 预览环境，并坚持：

- 从唯一候选 commit 构建；
- 生成包含 Git SHA、文件清单、总字节数和 SHA-256 的 manifest；
- Toy、GitHub、测试报告和通知都引用同一个 manifest；
- 工作树漂移后必须重新构建、重新验收、重新生成预览；
- 一个版本只有一份发布记录，历史报告不再复制过期指标。

## 13. 下一款游戏的正确制作流程

以下时间均是 CANOPY 类中小型生存游戏的**默认调参起点**，不是所有题材的普遍常数。每个阶段门使用冻结 build、固定或记录的 seed、明确测试者画像和标准观察表；除游戏内正式引导外，开发者不口头解释。默认至少 5 名目标玩家，关键理解与完成指标要求 `≥4/5`，并记录分子/分母、放弃点、主动计划数、最有趣/最无聊时段和继续/重玩意愿。连续两轮未过门必须回退到上一个阶段重构，不能靠增加教程或内容惯性前进。

### G0：命题与原创边界

交付：一句话玩家身份、环境压力、核心动词、目标情绪、参考游戏方法表、IP/资产边界和反目标；同时确定早期轮廓语言、对象比例、地标、夜间可读性和目标硬件预算。

门禁：团队能回答“为什么必须是这款游戏”，参考作品只拆方法，不复制角色、剧情、地图、文本和资产。

### G1：因果蓝图

交付：3–5 个体验支柱、资源—风险—反制链、三层循环、对象语义矩阵、最小经济表和唯一时间经济表。

门禁：每个核心选择默认改变两个系统；每种危险有信号和反制；日长、休息跳时、动作耗时、腐败、燃料和再生共享同一单位契约，并列出“现实分钟—游戏小时—主要生命周期”；暂不制作几十个物品和敌人。

### G2：3–10 分钟空间灰盒

交付：一个真实场景、一种资源、一种工具、一种威胁、一个世界后果和一次失败恢复。

门禁：至少 4/5 名首次玩家的首次有效行动不晚于 30 秒，三分钟内完成完整因果；抽样对象至少 90% 可预测动词和阻塞原因。四向视图能形成方向判断，至少记住一个地标、一个高差和两条可解释路线；大而平且空的地图不能用面积冒充探索内容。

### G3：24–30 分钟序章纵切

交付：一个群系、两条路线、营地安全机器、死亡/读档、明确结局和继续探索入口。

门禁：至少 4/5 名首次玩家不依赖外部 Wiki 完成；能说出两次真实取舍和直接死因；手机与桌面均具备全部核心动词。记录其主动形成的计划、放弃点、最有趣/最无聊时段和继续意愿；若只是“能通关但不想继续”，必须回到循环而不是扩内容。

### G4：30–65 分钟 A1 扩展段

交付：一个新生态事实、一次持久世界改变、一次路线/建造选择和一个新的准备问题。

门禁：至少 4/5 名目标玩家完成，实际墙钟进入预算；没有用重复交同一材料拉长时间；至少一次新世界事实让玩家主动改变计划。

### G5：65–105 分钟 A2 扩展段

交付：第二类群系问题、前哨网络、加工/保存和受限资源导演。

门禁：至少 4/5 名目标玩家完成；连续同类劳动默认不超过 12 分钟；关键资源既不保送也不形成随机死档；读档后世界状态一致。若单一最优解支配所有行动，回到经济和路线设计。

### G6A：A3 章节点

交付：25–45 分钟的新群系问题，包含准备选择、世界事实、持久改变和路线/建筑选择。

门禁：该节点独立通过 4/5 完成率、资源可达、持久化和趣味观察后，才能制作 A4。

### G6B：A4 章节点

交付：25–45 分钟的新移动或生存能力，并让旧营地/路线产生新的价值。

门禁：该能力不只是钥匙；至少支持两种策略，并在死亡、读档和回程后保持一致，之后才能制作 A5。

### G6C：A5、结局与三小时总测

交付：25–45 分钟的事实驱动终局、结局选择和继续沙盒。

门禁：逐章门禁全部通过后再做整局测试；首次玩家中位目标 2.5–3.5 小时，熟练玩家能靠知识缩短流程，至少两条策略可达。设计节拍位置、熟练玩家预计时间和首次玩家实测必须分栏记录。

### G7：正式视觉与性能

交付：程序化地形层次、群落构图、统一材质、光雾天气、音景、LOD、对象池和硬件预算。

门禁：G0–G3 已验证轮廓、比例、地标、夜间可读性和硬件底线；本阶段再收敛材质、群落、光雾天气和构图。每项变化提升可读性、情绪、身份或空间记忆，真实设备记录帧时间、内存、GPU、区块和包体曲线。

### G8A：Alpha 功能冻结

门禁：所有承诺系统可完成，存档和失败恢复成立，禁止再靠占位假系统；定义允许遗留的 P1/P2、崩溃率、恢复成功率和设备覆盖，不再新增破坏架构的功能。

### G8B：Beta 内容冻结

门禁：内容、引导、平衡、多端、性能、可访问性、导入导出和迁移全部经过首次玩家与真机矩阵；给出性能分位数、存档恢复率和剩余缺陷上限，除阻断问题外不再扩内容。

### G9：RC / 发布 / 线上复测

交付：冻结候选、回滚点、同一 commit 的 GitHub 与平台包、预览证据、审核状态、生产 smoke 和复盘入口。

门禁：不能用“构建成功”替代“可玩成功”，不能用“审核中”替代“已上线”。

### 14.4 内容关系检查

任何新增物品、动物、植物、建筑或群系必须回答：

1. 它改变哪两个以上已有系统？
2. 玩家怎样从轮廓、位置或声音认出它？
3. 它创造什么新选择，而不是只增加收集数量？

## 15. 下一款游戏开工后的 7 / 14 / 30 天计划

这是默认检查点模板，可按团队、引擎和题材风险调整日历；可以调整时间，不能跳过对应证据门。

### 前 7 天：只证明因果

- 完成命题、原创边界和反目标；
- 调研 3–5 款参考游戏，只记录机制、节奏、UI 和失败处理方法；
- 画出三层循环、资源—风险—反制链和对象矩阵；
- 做一个 3–10 分钟空间灰盒；
- 让首次玩家不看说明完成一次完整因果；
- 不做最终画面、不接发布、不扩几十种内容。

### 前 14 天：做序章纵切

- 加入一个营地/安全机器、一个危险和两条路线；
- 建立动作事务、稳定对象身份和最小存档；
- 建立桌面与触控的核心动词等价；
- 建立第一批不变量测试和浏览器 smoke；
- 至少完成两轮首次玩家观察和一次非实现者红队。

### 前 30 天：证明可扩展性

- 扩为 24–30 分钟完整序章；
- 加入第二种空间问题，而不是第二套换皮资源；
- 证明世界差量、恢复点、导入导出和生成器版本策略；
- 建立 CPU/GPU/内存/包体预算；
- 生成平台预览，但只有在玩法门禁通过后才进入正式发布准备；
- 根据真人数据决定继续扩章、重做循环或停止项目。

## 16. 每周十项检查

1. 玩家试玩后说的是“我读懂并解决了什么”，还是“我把哪些条补满了”？
2. 30 秒内是否有有效行动，三分钟内是否出现完整世界因果？
3. 随机抽查 10 个同类对象，至少 90% 是否能预测动词和阻塞原因？
4. 关键动作是否立即响应，并由动画、声音、世界变化和 UI 中至少三类共同确认？
5. 每种伤害是否有前兆、来源、状态升级、反制和死亡因果？
6. 新内容是否改变至少两个既有系统？
7. 是否存在连续超过 12 分钟的同类劳动或超过两个往返的单材料门槛？
8. 导演是否尊重地理、生态、进度、视野、公平和可复现性？
9. 建筑、采集、尸体、火、任务和知识能否经过存档、离区块和导入后保持一致？
10. 每项“完成”是否标明真实证据等级，且没有把自动化冒充玩家体验？

## 17. 全项目 Definition of Done

一项游戏能力只有同时满足以下条件才算完成：

- 玩家能在世界中发现它，而不是只在说明里知道；
- 玩家能预测正确动词或看到明确阻塞原因；
- 动作经过统一事务并产生具身反馈；
- 成功和失败都改变权威世界状态；
- UI、3D、音频和存档对同一事实一致；
- 键鼠和触控拥有等价的核心能力；
- 自动化覆盖规则、不变量和持久化；
- 浏览器或真机完成真实路径；
- 首次玩家证据与需求风险相称；
- 文档准确区分完成、部分和计划；
- 发布包与验收候选是同一版本。

## 18. CANOPY 仍未关闭的风险

这些内容必须保留为下一阶段任务，不能被本次发布和复盘掩盖：

- 尚无完整三小时或十小时首次玩家墙钟证据；
- 完整资源/威胁/章节导演仍未形成；
- 捕食、被捕食、食腐、植物压力和承载力回写仍未形成完整可观察生态链；
- A3–A5 章节仍未实装；
- 建造拆除与材料返还、储存、绳索移动、高脚/高平台前哨网络仍未形成完整纵切；
- Valheim 启发的正式地形、群落、材质、灯光、雾效与 GPU profiling 尚未闭环；
- 广泛真实手机、浏览器和性能矩阵仍不足；
- `worldGeneratorVersion` 尚未成为当前存档顶层身份，生成器升级后的旧档兼容未验证；
- 地形、河流、生态语义以及移动/固定步长同步边界仍有多真值风险；
- `GameClient`、主渲染器和模拟核心仍是高耦合热点；
- Toy 正式 URL 的首屏、既有存档恢复和游戏场景已通过；Pointer Lock 手感、云同步、跨设备恢复与广泛真机仍待复测；
- 当前 Toy 制品来自未完全提交的工作树，已有包清单指纹但没有与唯一 Git SHA 绑定；
- “规则可通关”已经有证据，“完整体验好玩”仍需要真实玩家证明。
