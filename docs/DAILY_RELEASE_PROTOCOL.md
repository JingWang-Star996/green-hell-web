# CANOPY 日更维护与发布账本协议

## 目标

日更不是每天堆一批素材，而是每天交付一条玩家能发现、能操作、能验证、能保存的最小纵切。每条纵切必须强化“可读的雨林、具身交互、持久后果”至少两项。

## 每轮顺序

1. 从反馈矩阵与执行积压选择一个可在同一天闭环的问题，不以任务数量代替可玩结果。
2. 协调器先写清输入、世界规则、反馈、失败原子性、存档边界和非目标，再按依赖派发模拟、UI、内容或红队子任务。
3. 先锁权威模拟和持久化测试，再接桌面/触控表现；同一游戏动词不得只存在于一种设备。
4. 在 `src/game/releaseNotes.ts` 最前方追加一个唯一 `YYYY.MM.DD.N` 候选条目。只有与 Toy 公开制品核对成功后才能把状态从 `candidate` 改为 `published`；已经公开的历史条目只读。公告只写有源码、测试或试玩证据的事实。
5. 玩家 Wiki 必须同步新增规则、限制、获得/使用方式和常见阻塞；游戏开始页与 Wiki 使用正式 Toy URL 双向跳转。
6. 执行 `npm run verify:release` 汇总代码、Toy 构建闭包、正式 Wiki URL/current buildId、Wiki 与宝典制品验证，再单独执行 Pages 构建并做桌面、竖屏与短横屏浏览器冒烟。
7. 记录仍未通过的人类判断闸门。自动化成功不得改写为“完整生态”“三小时内容”或“最终画面升级”。
8. 按游戏、玩家 Wiki、游戏创作宝典/发布证据的文件边界提交；推送后核对远端 SHA 与现有草稿 PR。

## 公告契约

- 最新条目永远在数组首位；同日多次发布递增末位序号。
- 同一时刻最多保留一个 `candidate`，且它只能位于数组首位；候选可在公开前纠正，公开后必须改为 `published` 并冻结完整标题与条目。
- 标题描述玩家能感受到的主题，变更条目按“新增/世界/多端/存档/修复/优化”分类。
- 一条公告对应一个可追溯构建，不使用“若干优化”“体验提升”等无法验收的占位语。
- 若候选构建没有公开，则可以在提交前补齐同一条目的真实内容；一旦公开，需记录制品摘要与 Toy 版本证据，后续修复必须新增版本，不能重写历史。

## 阻塞发布的闸门

- 核心动词在 PC 与触控任一端没有入口。
- 命令可产生部分结算、重复收益、错对象、不可恢复丢失或存档倒退。
- 开始页公告、Wiki 或实际代码互相矛盾。
- Toy 相对资源闭包失败、包体超限、浏览器出现阻断错误，或 Git 工作树混入临时服务日志/构建目录。

## 当前下一轮候选

- 火焰对动物的可读威慑与路径反馈。
- 营地命名/地图标记，支撑真正的多前哨管理。
- 安全扩展可拆设施范围（优先火把路标，必须保留每支火把的实体燃料所有权）。
- V1A 跨活动环实例池性能基线；通过后才推进 Valheim 启发的连续地形与材质层次。
