# CANOPY 生存 UI / UE 代码审计

审计日期：2026-07-14
审计对象：`green-hell-toy` 当前工作树
审计模式：只读代码审计；未运行浏览器、未修改游戏代码

## 1. 结论先行

当前版本已经不再是最初的“所有行为塞进一个菜单”：它拥有分类背包、制作/营地/建造/休息分组、五格快捷装备、世界内聚焦与阻塞解释、营火燃料条、烟熏进度、第一人称工具模型、手表夜光与火把。这些都应保留。

真正仍然阻断体验的，是三条链没有闭合：

1. **动作真值没有闭合。** 聚焦、提示、键鼠/触控输入和模拟器都知道一部分规则，但“能否执行、由什么输入执行、何时命中、何时结算”仍在多处重复判断。不可执行的“绕行/保持距离”和阻塞目标也会显示统一的 `E` 动作外观；采集、制作和休息则在一次同步调用中直接快进全部工作时间。
2. **反馈真值没有闭合。** HUD 日志、危险字幕、面板内反馈、音效映射和存档状态是五条独立通道。面板上层现在能显示打开面板后最新的一条事件，这是有效修复；但它仍只用“事件 ID 比打开/点击前大”猜测操作结果。一次命令产生多条事件时只显示最后一条，较早的危险副作用仍被模态层后的 HUD 日志遮住，任务、天气等后置事件也可能取代本次操作的主回执。
3. **设备真值没有闭合。** 世界主动作最终都能走到 `performCurrentAction()`，这是正确基础；但触控只能直接打开背包和身体，缺少制作、建造、地图、手表、笔记、暂停入口，手柄路径完全不存在。快捷键还出现了已经可见的文案漂移：实际为 `1–5`，标题页和背包仍写 `1–3`。

因此下一批不应先继续加菜单皮肤或物品数量，而应先建立一个端到端的 **Action Transaction（动作事务）+ Action Receipt（操作回执）+ Input Action（语义输入）** 小体系，再用“砍树/开采、添柴、休息”三个纵切验证它。这样后续动物、建筑和视觉升级才不需要各自再造一套交互。

## 4. P0：会破坏核心操作信任或遮蔽关键结果

### P0-1 操作回执会被遮住、误归因或永久滞留

证据：

- HUD 事件栈来自 `state.eventLog.slice(-30).reverse()`，始终渲染最新三条，没有“本次进入游戏后新发生”或过期时间。因此读档后历史事件也会像当前提示一样常驻。
- `.event-stack`、`.hazard-caption`、`.save-status-indicator` 的层级分别位于 HUD/`z-index:24`/`22`，而 `.panel-backdrop` 为 `z-index:30`。打开背包或身体面板时，这些反馈在视觉上处于模态层之后。
- `Panels.tsx` 会对所有面板显示打开后最新的一条新事件；只有制作按钮额外设置 `pendingAction`，而该状态目前只用于计算添柴前后值，没有为通用面板动作建立 command 关联。
- 一次工作快进可能同时产生腐败、熄火、天气、任务和制作事件，代码没有 transaction ID，也没有将回执与当前 command 关联。面板只取最新一条：例如冒险饮用浑水时，较早产生的 `parasite-contracted` 可能被随后追加的 `water-drunk` 覆盖，而完整 HUD 事件栈又位于模态层之后。危险事件还会被统一标成“操作未完成”，即使它只是成功操作产生的副作用。
- 回执没有清除策略；只要 `feedbackEvent` 仍在最近事件数组中，它就持续显示。
- 事件 tone 在 `viewModel.ts`、音效 cue 在 `GameClient.tsx` 分开硬编码。新增的 `wildlife-attack` 等事件未被 tone 表识别为危险，颜色、声音和伤害闪烁可能表达不同严重度。

影响：玩家无法可靠知道“按钮有没有生效、刚才发生的后果属于什么、是否需要立即处理”。这正是此前“提示被 UI 盖住”的根因，只修了制作面板的一个分支。

最小修复原则：建立单一的 `ActionReceipt`，不要继续给每个面板加一块特例 toast。

### P0-2 交互提示把“可观察”画成“可执行”，并在多处重复判定

证据：

- `Hud.tsx` 对任何 target 固定渲染 `<kbd>E</kbd>`。
- `TouchControls` 的 `actionAvailable` 来自 `placementRecipe !== null || target !== null`，不是 affordance 是否可执行。
- renderer、`commandForInteraction()` 各自重复一遍“ready，或 danger+attack 才可执行”的规则；但 renderer 即使判断为不可执行，仍会把 target 回调给 React，由 React 再显示说明文字。
- 无矛的蛇/捕食者把主动作写成“绕行/保持距离”，但该动作没有 command；按 E 或触控按钮不会执行绕行动作，只会弹说明。
- blocked target 也使用相同 E 外观。玩家无法在操作前区分“按下会执行”“按下只会解释”“应该靠移动完成”。

影响：这是统一交互语言中的“假按钮”。对于树、岩石、动物这种高频目标，会快速消耗玩家对准星提示的信任。

最小修复原则：由 affordance 输出单一 `disposition`：`execute | inspect | movement | unavailable`，同时输出主输入语义；HUD、触控、键鼠和 renderer 只投影它，不能再自行猜测。

### P0-3 玩家动作没有事务、命中窗口或重复输入保护

证据：

- `advanceWorkTime()` 在当前 JS 调用内循环全部固定 tick。采集 2–5 秒、调查 12 秒、建筑 18–90 秒、休息八个游戏小时都在界面获得下一次绘制机会前完成。
- `performCurrentAction()` 先让持有模型播放 0.34 秒动画，再同步派发已完成结算；模拟资源减少、耐久损耗和产出并不等待动画命中。
- 只有 `tool.*` 和 `weapon.*` 会触发持有动画，`hand.pickup`、`hand.add-fuel`、`hand.inspect`、`player.rest` 等 animation key 没有对应执行表现。
- 没有 `player.action`、cooldown、input buffer 或一次性 settlement token。连续点击可在前一次摆动尚未结束时重复结算。
- affordance 的 `estimatedSeconds` 不完整：砍伐常有值，开采/切割/拾取多为 `null`；HUD 也完全不显示它。

影响：第一人称工具只是结算后的装饰，采集与战斗仍是“点击数字变化”。这也是工作进度、战斗动作感、砍树反馈和营火动作无法自然扩展的共同根因。

最小修复原则：不要一次重写所有动作。先用下一次树木/岩石纵切建立一个保存兼容的 `PlayerActionState`，证明开始—预备—命中—恢复—完成/取消的闭环，再移植到其他动词。

## 5. P1：下一轮应系统性补齐的体验

### P1-1 休息“关闭面板”已修，但缺少睡眠转场与结果摘要

现状值得保留：菜单休息成功后关闭 UI，床也支持世界内直接休息，休息确实结算八个游戏小时、天气、火、腐败、伤口和代谢。

仍需补齐：

- 接受休息后立即锁定输入并收起面板。
- 先绘制黑场/闭眼转场，再开始隔离的时间快进；不要让同步循环阻止转场出现。
- 完成后展示一张 2–3 秒可跳过摘要：`14:00 → 22:00`、生命/水分/能量变化、营火是否熄灭、哪些食物腐败、是否发生危险。
- 如果快进导致死亡，直接进入死亡结算，不能先显示“休息成功”。
- 摘要结束后才恢复观察输入；自动存档状态在摘要中可见。

### P1-2 营火数字已经正确，但世界没有响应添柴

当前本地 E、燃料条、满燃料保护和面板前后值都是正确基础。缺口是“世界先反馈，UI 再解释”：

- 营火焦点代理携带 fuel preview，但世界火焰/光照更新只直接读取 `fireLit`；火光强度不随 fuel ratio 变化。
- 添柴没有手部/木料进入火堆动作。
- 火堆内没有可由燃料档位驱动的木柴数量或焦黑状态。
- `fuel-added` 没有独立音效 cue，也没有一次性火星/火焰冲高。

建议把添柴作为第二个动作事务纵切：6 秒游戏时间可用约 0.6–0.9 秒真实动作表现，命中点才消耗木棍并增加燃料；成功后木柴、火焰/光、火星/声音、燃料条和短回执同步变化。

### P1-3 制作、建造和本地维护仍没有形成清楚的信息架构

虽然 `C` 面板已经分组，但它仍同时承担：随身合成、煮水/接雨/添柴、自由放置建筑、修复电台和休息。玩家仍会形成“所有事先去 C 里找”的习惯。

建议的主表面：

- **C / 制作**：只保留随身工具、药品、容器、食物处理配方。
- **V 或独立建造入口**：只保留可放置结构，进入蓝图/放置模式；触控使用一个“建造”入口。
- **世界对象 E**：营火添柴、床休息、烟熏架装载/收取、信标发送等本地行为。
- **长按 E / 上下文**：后续才加入选择燃料、烹饪、拆除等多动作；当前批次不必先造完整径向菜单。

当前还有四处重复真值：

1. `RECIPES[recipe].effect` 决定真实效果。
2. `actionUx.ts` 的 `WORLD_BUILD_ACTIONS` / `CAMP_ACTIONS` 决定 UI 分组与关闭。
3. `GameClient.tsx` 的 `BUILD_RECIPE_IDS` 决定是否进入放置。
4. `Panels.tsx` 的 `craftingActionLabel()` 决定按钮文案，而且漏掉了 `smoking-rack` 的“搭建”。

应把 `section`、`surface`、`actionLabel`、`closePolicy`、`placementKind` 放入一份配方表现元数据，或从 recipe effect 纯函数推导，禁止再加平行 ID 集合。

### P1-4 输入需要一个语义路由器，而不是继续加按键分支

当前优点：E、鼠标、触控世界动作最终共享 renderer 主动作入口；装备按钮和背包装备共享模拟 command。

当前缺口：

- 键盘面板/装备映射在 `GameClient`，移动/E 在 renderer，触控在 React 组件；没有统一 `InputAction` 层。
- HUD 固定显示 E，不知道玩家最近使用的是鼠标、触控还是手柄。
- 触控完整交互详情被 CSS 隐藏，只剩动词和截断目标名；燃料、生命、烟熏进度、工具阻塞详情都丢失。
- 触控没有制作、建造、手表、地图、笔记、暂停入口，无法完整完成主线。
- 仓库中没有 `navigator.getGamepads()` 或任何手柄映射。当前不能声称支持手柄。
- 标题页、背包装备说明写 `1–3`；实际映射与快捷栏是 `1–5 + Q`，既有文案漂移已经证明硬编码会继续出错。

建议建立 `InputAction`：`move`、`look`、`primary`、`use`、`cancel`、`equip-slot-1..5`、`stow`、`open-inventory`、`open-craft`、`open-build`、`open-watch`、`open-map`、`open-notebook`、`pause`。设备适配器只生成这些动作，UI glyph 也从同一映射投影。

桌面仍为首要平台；手柄可先做最小 Xbox/标准 Gamepad 映射和设备切换提示，不必立即做全品牌图标包。触控作为声明的兼容路径，则必须至少能完成主线。

### P1-5 背包分类可用，但“紧急”与玩家真实需求脱节

当前 `isUrgentInventoryItem()` 会把所有可食用、可饮用、可使用药品判为紧急，即使玩家状态满、食物稳定。因此“紧急相关”很容易成为另一个“全部”，长期会让警告失去意义。

建议让 view model 输出明确的 `urgencyReason`，而不是让 UI 只看 item category：

- `spoils-soon`：临期阈值。
- `need-hydration` / `need-energy`：玩家对应状态低于阈值且物品能解决。
- `treats-current-condition`：当前确有伤口/寄生虫。
- `equipped-low-durability`：当前手持工具将坏。
- `mission-required-now`：当前目标需要。

本批次只需修正紧急排序、明确当前已装备状态和 `1–5` 说明。重量、长材槽、搜索和空间化背包可以放到 P2，不应挡住动作闭环。

### P1-6 第一人称反馈需要从“有模型”升级为“动作有因果”

保留五种持有模型与 bob。下一步不需要先做高成本动画资产，而应统一最小动作词汇：

- 空手：伸手拾取、放置/装载、检查。
- 斧/镐/刃：预备、挥动、命中反冲、恢复；不同工具至少有方向和节奏差异。
- 矛：预备、突刺、命中/落空恢复，期间不能无限重触发。
- 火把：占手状态明显；切换到双手/冲突装备时给出明确结果。
- 受击：屏幕闪烁之外，加入方向性提示；危险字幕与伤害事件使用同一 severity。

动作动画只负责表现，模拟器仍负责结算；但模拟器必须等待经过验证的命中时刻，而不是在动画之前完成所有结果。

## 6. P2：在 P0/P1 纵切稳定后再做

### P2-1 背包容量与长材管理

- 当前只有每种物品 stack limit，没有总重量/体积/长材槽反馈。
- 先增加容量摘要和拾取去向，再考虑捆扎、长木段槽和搜索。
- 不建议现在做自由拖拽像素整理；它会放大触控和存档复杂度，却不解决核心生存决策。

### P2-2 UI 可读性与缩放

- CSS 大量使用 6–8px 文本，装备耐久甚至为 6px；在高 DPI、Toy iframe 或缩放页面中可能不可读。
- 需要浏览器恢复后按 1366×768、1920×1080、窄屏触控三档检查，之后增加 `100% / 115% / 130%` UI scale。
- 所有关键状态不能只靠颜色；当前大多已有文字，可继续保持。

### P2-3 夜间可访问性与实际视觉门槛

代码已经提供自动手表光和更强火把，不应重做。后续只需：

- 增加受限的“夜间可读性”设置，调整最低环境轮廓/眼适应，而不是把夜晚变白天。
- 在触控也显示当前光源与剩余燃烧信息。
- 恢复浏览器后走固定夜间路线，记录是否能辨认地面、可交互轮廓、返营方向与蛇预警。

纯函数只能证明“光源参数存在”，不能证明玩家实际看得见，因此该项不能仅凭单元测试宣告完成。

### P2-4 手柄细化与触觉

最小标准 Gamepad 路径进入 P1；品牌 glyph、灵敏度曲线、震动强度、南键/东键地区习惯切换、长按上下文等可在 P2 处理。

## 7. 重复真值清单

| 领域 | 当前重复位置 | 已出现的实际漂移 | 应收敛到 |
|---|---|---|---|
| 动作可执行性 | affordance、renderer、`commandForInteraction`、Touch `target !== null` | blocked/avoid 仍显示可按动作 | `resolveActionDisposition(affordance)` 纯函数 |
| 操作结果 | event log、`lastEventIdRef`、面板 pending event、hazard caption | 无关新事件可冒充当前回执；面板动作结果不可见 | simulation 产生带 transaction ID 的 `ActionReceipt` |
| 事件严重度 | `toEventView()` tone、GameClient 音效/闪烁 if 链 | wildlife damage 可显示中性色 | `eventUxProfile(type)` 单表，驱动文字、颜色、声音、震动 |
| 配方表面 | recipe effect、两个静态 ID Set、按钮文案函数 | 烟熏架落到“制作”文案 | 单一 recipe presentation metadata |
| 快捷装备 | GameClient 数字映射、EquipmentBar 数组、标题/背包文字、规格文档 | 实际 1–5，文案 1–3/1–4 | 单一 equipment slot registry |
| 工作时长 | simulation 常量、recipe `workSeconds`、affordance `estimatedSeconds`、UI 材料文本 | 多个动作估时为空；表现 0.34 秒而结算 2–90 秒 | action definition + transaction state |
| 消息生命周期 | durable event log 同时充当 HUD toast | 读档历史常驻 HUD | 持久日志与 session notification queue 分离 |

## 8. 建议的下一批最小纵切与顺序

### Slice UX-0：全局操作回执（P0，先做）

目标：任何面板和世界动作都有一个不会被遮挡、不会误归因、会自动消失的回执。

最小范围：

1. 为离散 command 分配 transaction ID；模拟输出 `accepted/rejected/completed/interrupted` 和主结果事件 ID。
2. 新建全局 `ActionFeedbackLayer`，层级高于普通面板、低于胜负/兼容性对话框；不抢焦点。
3. 把 crafting 专用 `pendingAction`、普通事件 toast 和危险字幕纳入统一队列的不同 lane。
4. 读档历史只进入笔记日志，不进入 session toast。
5. 事件表现表统一 tone、音效、伤害闪烁和可选触觉。

自动化验收：

- `tests/ui/actionReceipt.test.ts`：同一 work interval 中插入天气/腐败/任务事件，回执仍指向发起 command 的结果。
- 背包饮水导致寄生虫、身体治疗失败、制作成功、燃料满拒绝都能生成唯一回执。
- 加载带旧 event log 的存档时 session 通知队列为空。
- 回执超过 TTL 后移除；danger 类需要确认的消息按策略保留。
- SSR 检查反馈层位于 Panels 之后且有 `role=status/alert`；CSS 静态检查其层级高于 panel 30、低于 resolution 60。

### Slice UX-1：树木/岩石动作事务（P0，接当前树木纵切）

目标：用一次砍伐或开采证明“提示—输入—进度—动画—命中—世界结果—回执”是一条链。

最小范围：

1. 新增可持久化/可迁移的 `PlayerActionState`：target、verb、phase、startedTick、hitTick、completeTick、cancelPolicy、transactionId。
2. affordance 输出 disposition、trigger 和完整 estimated time。
3. HUD/触控显示短进度；blocked 显示“查看条件”而非假执行键；movement 型绕行不显示 E。
4. 输入开始动作，命中 tick 才扣耐久/目标生命并产生掉落；恢复期拒绝重复结算。
5. 移动过远、目标消失、切换装备时按明确规则取消，未命中不产出。

自动化验收：

- 开始动作后、命中前，目标 quantity、背包、工具耐久均不变。
- 到命中 tick 只结算一次；连续 E/鼠标/触控不能双结算。
- 命中前离开范围取消且不产出；命中后取消不回滚已发生结果。
- 键盘、鼠标、触控适配器对同一目标产生同一 `InputAction` 和 command。
- blocked/inspect/movement/execute 四类 affordance 的键位、按钮可用性和文案快照一致。

### Slice UX-2：营火添柴的六通道反馈（P1）

目标：添柴后 1 秒内，玩家不看日志也知道木柴已经加进去了。

最小范围：

1. RenderSnapshot 暴露 fuel ratio / fuel stage 和一次性 fuel-added presentation event。
2. 手部放柴；命中点消耗木棍。
3. 火堆木柴档位、火焰/光照短脉冲、火星、添柴音效同步。
4. 聚焦燃料条平滑上升并显示增加的游戏时间。
5. C 面板不再作为添柴主入口；世界 E 为主，菜单最多提供教学指向。

自动化验收：

- `resolveCampfirePresentation(fuel, capacity, lit)` 对 0/低/中/满四档稳定。
- 满燃料不消耗木棍，也不播放“成功添柴”表现。
- 成功 transaction 同时产生 inventory delta、fuel delta、render impulse、audio cue、receipt 五项；每项只发生一次。
- 保留现有“余燃料重燃不浪费木棍”回归。

### Slice UX-3：休息转场与睡眠报告（P1）

目标：玩家明确看见“睡了多久、付出了什么、醒来时发生了什么”，且休息后 UI 必定关闭。

最小范围：

1. `start-rest` 成功后立即关闭面板并进入 `RestTransition`。
2. 先绘制黑场，再在隔离状态中分块快进；完成后一次提交结果。
3. 用纯函数比较 before/after，生成时间、生命、水分、能量、火、腐败、天气/危险摘要。
4. 摘要结束后恢复输入并显示保存状态；死亡走胜负层。

自动化验收：

- `craftingActionPolicy` 之外增加协调层回归：accepted rest => panel null + transition active；rejected rest => panel 保持且有回执。
- 快进前黑场状态先提交一次，避免同步计算吞掉转场首帧。
- `summarizeRestOutcome()` 覆盖跨日、熄火、腐败、受伤、死亡。
- 摘要的 end time 与模拟器 clock 完全一致；autosave reason 仍为 `rest`。

### Slice UX-4：输入语义与触控主线完整性（P1）

目标：同一个游戏动作只有一个语义，不同设备只是不同映射。

最小范围：

1. 建立 input action registry 与 equipment slot registry。
2. 修复所有 `1–3/1–4/1–5` 漂移，UI glyph 由最近活动设备决定。
3. 触控增加一个紧凑菜单入口，覆盖制作、建造、手表、地图、笔记、暂停；完整 target detail 用可展开卡展示。
4. 标准 Gamepad 最小映射：双摇杆、南键使用、右扳机主动作、肩键换装备、菜单键暂停；没有设备时不渲染手柄提示。

自动化验收：

- `tests/ui/inputParity.test.ts` 列出完成主线所需动作，keyboard/mouse、touch、gamepad 每列都必须有映射。
- EquipmentBar 渲染键位、GameClient 路由与标题说明全部读取同一 registry。
- touch target 对 blocked/ready/danger 的按钮状态和详情与 HUD 相同。
- 面板打开、页面隐藏、指针锁释放时所有持续输入归零，防止移动/冲刺卡住。

### Slice UX-5：配方表面与背包紧急性收敛（P1/P2）

目标：扩内容时不再出现分类漂移，并让背包警告真的有优先级。

自动化验收：

- 每个 `RecipeId` 恰好拥有一个 presentation profile；placeable effect 必须进入 build surface。
- `smoking-rack` 显示“搭建”，本地维护行为不混入随身制作。
- 满状态玩家的稳定食物不进入紧急筛选；低水分时净水进入并携带原因；无寄生虫时驱虫药不紧急。
- 任一已有物品类别两次操作内可达；五种装备无需打开背包即可切换。

## 9. 自动化门槛总表

| 门槛 | 可自动化证据 | 仍需人工/浏览器证据 |
|---|---|---|
| 回执不误归因 | transaction/receipt 纯函数测试，事件乱序与并发系统事件测试 | toast 位置、时长、注意力负担 |
| 面板不遮反馈 | 组件 SSR 顺序与 CSS 层级契约测试 | 实际模态、缩放和 Toy iframe 遮挡 |
| 动作只结算一次 | deterministic action-state tests | 动画手感、命中重量感 |
| 休息关闭与快进 | UI coordinator reducer + simulation clock tests | 转场节奏、摘要停留时长 |
| 营火多通道反馈 | snapshot/presentation/audio event tests | 木柴、火焰、光和火星是否自然 |
| 输入一致 | input action matrix tests | 触控灵敏度、手柄死区、误触率 |
| 背包查找效率 | 分组/筛选/registry exhaustive tests | 真实玩家是否能在两次操作内找到目标 |
| 夜间底线 | light profile 阈值和 HUD 状态测试 | 固定夜路可读性、屏幕/环境差异 |

## 10. 本轮不建议做的事

- 不先重画整套 UI；当前最大问题是状态与反馈协议，不是主题色。
- 不给每个面板各写一个 toast；这会把当前重复真值扩大。
- 不在没有 action state 的情况下继续添加更多工具摆动；动画数量不会修复提前结算。
- 不马上做自由拖拽、复杂径向菜单或品牌手柄 glyph 包；先保证核心动作和主线完整。
- 不用单元测试替代夜间与触控试玩；浏览器恢复后必须补固定路线的实际验证。

## 11. 完成判定

这一批 UI/UE 升级完成，不是“菜单看起来更丰富”，而是满足以下可观察结果：

1. 玩家看到提示时，能在五秒内判断它是可执行动作、查看条件还是应通过移动完成的应对。
2. 一次输入最多结算一次；世界结果发生在可见命中/完成时刻，而不是动画之前。
3. 无论世界、背包、身体还是制作面板，成功、失败和危险后果都不会被遮挡，也不会被无关事件冒充。
4. 添柴无需打开总制作菜单，且木柴、火焰/光、声音、燃料 UI 和回执同时响应。
5. 休息接受后立即关闭 UI，醒来后明确展示时间与代价。
6. 键鼠和触控都能完成完整主线；手柄只有在最小动作矩阵通过后才标为支持。
7. 夜间近距导航已经有代码底线，但必须等浏览器恢复后通过固定路线试玩，才能宣告视觉完成。
