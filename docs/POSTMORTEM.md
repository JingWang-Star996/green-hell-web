# 《绿色地狱》网页版首版复盘与取证报告

> 状态：首版失败复盘，作为重制立项的基线证据  
> 审计日期：2026-07-13  
> 审计对象：`green-hell-toy` 当前源码、配置、资源、依赖与既有静态产物  
> 原则：不追责个人，只追踪产品决策、流程缺口与可复现证据

## 1. 执行结论

当前成品的准确类别是：

> 一张《绿色地狱》主题的营销落地页，附带一个极简回合制按钮 Demo；它不是《绿色地狱》的网页版游戏，也没有覆盖原作最小可识别的体验骨架。

它的桌面视觉壳具有一定完成度，但没有形成空间、观察、采集、制作、身体诊断、环境威胁、成长和目标之间的闭环。玩家开始后只需搜集一次、生火一次，约 20–30 秒就耗尽全部新鲜内容；其后只剩重复点击和四个数字涨跌，而且没有胜利状态。

这次失败的根因不是“功能还不够多”，而是从一开始就选错了产品原型：团队先做了一个“看起来像游戏”的网页，再向里面塞按钮；没有先证明一个可玩的生存循环。

## 4. 当前实际实现了什么

### 4.1 状态与内容

- 4 项状态：生命、体力、理智、水分，见 [`app/page.tsx:5-7`](../app/page.tsx#L5-L7)。
- 4 种物品：木棍、果实、草药叶、石片。
- 6 个全局按钮：搜集、水源、检查、生火、休息、探索，见 [`app/page.tsx:34-40`](../app/page.tsx#L34-L40)。
- 3 个概率分支：
  - 搜集获得果实：58%；
  - 喝水染寄生虫：28%；
  - 探索受伤：38%。
- 伤口、寄生虫、营火三个布尔状态。
- 回合消耗、简化昼夜、失败弹窗和分数。
- 身体检查弹窗、背包/日志/守则页签。
- LocalStorage 与 Toy 云存档意图。
- 响应式 CSS 和 `prefers-reduced-motion`。
- Toy `reportAction` 只上报“开始游戏”，见 [`app/page.tsx:215`](../app/page.tsx#L215)。

### 4.2 没有实现什么

没有地图、移动、角色、场景实体、拾取节点、配方树、制作界面、工具、武器、耐久、建造、庇护所、烹饪、狩猎、捕鱼、动物 AI、战斗、环境交互、完整天气、宏量营养、感染发展、毒素、发烧、蛭虫、部位伤势、故事、NPC、章节、难度、胜利条件、成就或有意义的重玩变化。

## 5. 可复现的实际核心循环

当前循环是：

```text
开始
→ 点击一个全局按钮
→ Math.random 决定一句日志
→ 四个数字改变
→ 死亡或无限重复
```

浏览器实测步骤：

1. 尚未开始时，界面已经给玩家 212 分；营火目标进度已是 67%。
2. 点击“开始求生”。
3. 当前只有 2 根木棍，但标注“消耗 3 根木棍”的营火按钮仍然可点击。
4. 点击失败后，唯一结果写入默认不可见的“日志”页签；主画面没有即时错误反馈。
5. 点击一次“搜集物资”，必定增加 1 根木棍和 1 片草药叶。
6. 点击“生起营火”。此时仅经过两回合，时间为 16:00。
7. 目标变成“熬过第一个雨夜”，进度却因木棍已被消耗而显示 0%。
8. 新目标没有完成条件，整个程序没有胜利状态。

初始 212 分可由代码直接还原：

```text
day * 180 + turn * 35 + inventoryCount * 8
= 1 * 180 + 0 + (2 木棍 + 1 果实 + 1 叶片) * 8
= 212
```

对应公式见 [`app/page.tsx:100-104`](../app/page.tsx#L100-L104)。玩家还没有做任何事就获得分数，说明分数不是表现评价，只是不断上涨的装饰数字。

一个最小可识别的生存循环应是：

```text
观察身体与局部环境
→ 选择空间目标和可承担风险
→ 移动、识别、采集特定资源
→ 组合材料制作工具、药物或庇护
→ 用制作结果解决当前威胁
→ 时间、天气和身体产生新后果
→ 解锁地点、知识、故事或长期目标
```

当前只实现了“随机改变数字”这一小段。

## 6. 十大严重问题

### P0-1：产品形态选错——做成落地页而不是游戏

证据：品牌栏、巨型宣传标题、宣传文案、三条法则和 footer 占据主体；实际行动区位于固定 610px Hero 之后。结构见 [`app/page.tsx:198-267`](../app/page.tsx#L198-L267)，布局见 [`app/globals.css:26-65`](../app/globals.css#L26-L65)。

影响：玩家的首要行为是阅读营销内容，不是观察世界和作出决策；后续再加按钮也不会自动变成游戏。

### P0-2：完全缺少空间和可交互对象

证据：六项行动均为无地点、无距离、无工具条件的全局函数分支，见 [`app/page.tsx:132-166`](../app/page.tsx#L132-L166)。“深入雨林”只运行一次随机数。

影响：玩家从未身处雨林；没有路线、方位、记忆、回程风险、资源分布或环境学习。

### P0-3：核心目标在约 20 秒内耗尽

证据：初始有 2 根木棍，搜集固定 `stick + 1`，营火只需 3 根，见 [`app/page.tsx:24-25`](../app/page.tsx#L24-L25)、[`app/page.tsx:138-155`](../app/page.tsx#L138-L155)。生火后的“熬过雨夜”只是一行文本，见 [`app/page.tsx:218`](../app/page.tsx#L218)。

影响：没有中期目标、内容展开、技能掌握、胜利或结局。

### P0-4：文案宣称的系统并不存在

证据：[`app/page.tsx:263`](../app/page.tsx#L263) 声称管理蛋白质、脂肪、碳水和水分，实际状态只有生命、体力、理智、水分。伤口文案称会感染，代码没有感染状态、计时器或伤口持续伤害。

影响：界面制造深度错觉，玩家一旦操作就会发现所有名词没有机械含义，信任快速崩塌。

### P0-5：资源和数值没有形成经济系统

证据：所有参数硬编码在组件中，搜集固定产木棍和叶片，水无限，营火永久，石片无任何消费路径，见 [`app/page.tsx:119-166`](../app/page.tsx#L119-L166)。

影响：不存在稀缺性、路线差异、配方权衡、准备成本和风险回报。数值只能拖延点击，不能制造策略。

### P0-6：伤病、寄生虫和营火都是“假系统”

证据：

- `wounded` 不造成失血、感染或行动惩罚。
- 寄生虫一旦出现无法治疗，弹窗只显示“检测到”，见 [`app/page.tsx:275-276`](../app/page.tsx#L275-L276)。
- 营火永不熄灭、不耗后续燃料，而且已点燃后仍可重复生火刷理智。
- 检查、治疗和吃果实都不消耗回合，见 [`app/page.tsx:168-181`](../app/page.tsx#L168-L181)。
- 吃果实、检查和包扎可在按下“开始”之前执行。

影响：系统名词存在，但没有相互作用、演化或决策重量。

### P0-7：行动缺乏即时、感官和因果反馈

证据：多数结果只写入日志，日志默认隐藏在另一个页签，见 [`app/page.tsx:114-117`](../app/page.tsx#L114-L117)、[`app/page.tsx:245-257`](../app/page.tsx#L245-L257)。没有行动动画、角色反馈、采集对象变化、音效、镜头、受击或环境响应。

影响：玩家感受不到自己“做了什么”，只能对比数字；这是表单反馈，不是游戏反馈。

### P0-8：界面和状态存在多处直接矛盾

证据：

- [`app/page.tsx:238-240`](../app/page.tsx#L238-L240) 计算资源不足，却只设置 class，没有绑定到 `disabled`。
- 生火消耗木棍后，新目标进度仍按木棍数量计算，实测从 67% 变为 0%。
- 文案写左臂，图形永久把右臂标红，见 [`app/page.tsx:273-274`](../app/page.tsx#L273-L274)。
- 伤口治好后，红色右臂和感叹号仍存在。
- 状态点只读取 `wounded`，不反映寄生虫、脱水或理智危机。
- “继续求生”再次点击没有游戏行为。

影响：玩家无法信任 UI，资源门禁和状态诊断均失去信息价值。

### P1-9：没有可扩展的游戏工程架构

证据：状态、规则、随机数、内容、存档、Toy 副作用和所有 React 视图耦合在一个组件中；没有 reducer/状态机、领域层、内容数据层、事件队列、种子 RNG、模拟器或版本化存档。

此外，存档读取虽然对 `applySave` 做了 `try/catch`，但时间戳的 `JSON.parse` 位于保护之外，见 [`app/page.tsx:60-85`](../app/page.tsx#L60-L85)；存档也没有 schema 校验。

影响：每增加一个系统都会扩大组合状态和回归风险，无法由多个 Agent 安全并行。

### P1-10：测试、构建和发布链不成立

证据：Lint 无配置、测试文件不存在、Windows 构建脚本直接失败、依赖树不干净、认证模块未使用、favicon 仍是脚手架默认图标。Toy SDK 以同步协议相对脚本放在 `<head>`，见 [`app/layout.tsx:14`](../app/layout.tsx#L14)。

影响：无法证明同一提交可重建、可回归、可发布，也无法将线上问题定位回规则或制品。

## 7. 分维度诊断

### 7.1 交互

- 只有点击，没有移动、指向、拖拽、组合、瞄准、选择身体部位或对象上下文。
- 按钮始终代表全世界可用的动作，不受地点、工具、光照、姿态和目标限制。
- 资源不足的按钮仍可点击；无效动作反馈藏在日志。
- 弹窗没有 Escape 关闭、焦点约束或动作后自然回到世界的流程。

### 7.2 反馈

- 主要反馈是数值和一行日志。
- 没有“意图预览 → 执行动作 → 世界响应 → 代价结算”的完整反馈链。
- 没有声音，CSS 雨、雾、脉搏和火焰只是持续装饰，不表达玩家行为。
- 没有死亡原因分解、资源变化明细和决策复盘。

### 7.3 视觉

优点：墨绿、米白、橙色的色彩统一；HUD、数值条、日志和身体检查有一致语言；桌面宽屏构图尚可；有响应式断点和 reduced-motion。

缺口：没有一张真正用于游戏的雨林、角色、生物或物品素材；背景是渐变、几何树冠和斜线雨；物品是 CSS 几何图形；宣传海报的真实感远高于游戏本身，放大了预期落差。

### 7.4 内容

当前内容容量只有 6 个动作、4 种物品、3 个随机表、1 个伤口、1 个寄生虫、1 个营火目标和少量日志分支。它不足以支撑一分钟的发现，更不足以支撑生存游戏的学习曲线。

### 7.5 数值

- 时间每 8 回合切一天，小时只在 14:00–21:00 循环；没有午夜和清晨。
- 黑夜实际上只有 19、20、21 点三个回合。
- 生命或理智归零才失败；体力和水分归零本身不会结束游戏。
- 没有数据表、调参记录、资源流模拟、死亡分布或策略测试。
- `Math.random()` 无种子，问题不可复现，自动平衡不可行。

### 7.6 技术债

- 单组件巨石结构阻止多人/多 Agent 并行。
- 内容和逻辑不可数据驱动。
- 存档无版本、无 schema、无迁移。
- 没有错误边界、事件追踪、领域测试或 E2E。
- 构建脚本不跨平台。
- 依赖与脚手架残留未清理。

## 8. 可以保留什么

以下内容可以作为重制参考或外壳继续使用，但不能把新玩法继续堆进当前组件：

- Toy 云存档和事件上报的接口意图。
- Next 静态导出的部署方向。
- 墨绿/米白/橙色视觉 token、字体层级和基础 HUD 卡片。
- 身体检查作为核心交互概念。
- 日志、背包、身体状态的中文文案基调。
- `clamp` 等少量纯函数。
- 非官方作品与商标归属声明。

预计现有 CSS 中约 30%–40% 可抽成 token、HUD 和通用控件；这不等于现有页面可继续作为游戏主体。

## 9. 必须重写什么

- 整体重写 `app/page.tsx` 中的游戏状态、动作、数值、目标、存档和主体交互。
- 建立独立世界模型、实体系统、物品/配方数据、事件队列、身体模拟、AI、任务和胜负状态。
- 将随机数改为可注入的种子 RNG。
- 建立版本化、schema 校验、可迁移、可恢复的存档。
- 将营销入口和真实游戏场景分离；进入游戏后以世界和可交互对象为主。
- 建立素材、动画、音频和反馈管线。
- 重建跨平台脚本、Lint、单元测试、模拟测试、E2E、CI 和发布流程。

## 10. 重制版量化验收门槛

### 10.1 核心体验门槛

- 新玩家 3 分钟内完成一次“观察 → 采集 → 制作 → 使用 → 世界后果”闭环。
- 垂直切片单局有效游玩时间不少于 15 分钟。
- 至少有 3 条可行生存策略；任何单一行动不得占全部行动的 35% 以上。
- 至少有 1 个明确胜利目标、4 类死亡原因和可理解的失败复盘。
- 不允许存在可无限净收益、永久无代价或无条件重复的动作。

### 10.2 垂直切片内容门槛

- 1 个可移动、可探索、可返回的完整生物群落。
- 至少 12 个空间资源节点、20 种物品、15 个有效配方。
- 至少 5 种工具/武器，具有耐久、用途和制作成本差异。
- 至少 6 种伤病、寄生或中毒状态，以及对应诊断和治疗路径。
- 至少 3 种动物或敌对 AI。
- 完整昼夜、天气、营火、庇护、烹饪和宏量营养闭环。
- 至少 3 个阶段目标和 1 个 15–30 分钟可达成结局。

### 10.3 玩家验证门槛

在至少 20 名不参与开发的目标玩家盲测中：

- 70% 以上对“好玩/愿意再玩”给出 4–5 分。
- 60% 以上完成新手闭环。
- 50% 以上主动开启第二局。
- 80% 以上能说出至少两项决策之间的实际取舍。
- 首局中位有效时长不少于 15 分钟。

### 10.4 反馈与性能门槛

- 输入后 100ms 内出现视觉响应，300ms 内出现明确结果反馈。
- 每个关键动作至少同时具备动画、声音、世界变化、数值解释中的三项。
- 中端桌面 60 FPS p95，目标移动设备至少 30 FPS。
- LCP 小于 2.5 秒，运行时无未处理异常。
- 关键 HUD 在 1280×720、1920×1080 与主流移动视口均不遮挡主要交互。

### 10.5 工程门槛

- `build`、`lint`、`typecheck`、单元、集成、E2E 在 Windows 与 CI 全部通过。
- 领域规则分支覆盖率至少 80%。
- 每个候选平衡版本至少运行 1000 局种子化自动模拟。
- 存档 schema 具有版本、迁移、校验和损坏回退。
- 每次发布记录胜率、平均局长、死亡原因、配方使用率、行动分布和第二局启动率。
- 发布必须能追溯到 commit SHA、构建编号、内容版本和验收记录。

## 11. 最终复盘

这次交付从“先做一个看起来像游戏的网页”开始，所以最终得到的是网页。正确顺序必须是：先研究并定义原作中不可替代的体验支柱，再用纸面和灰盒证明空间化核心循环真的好玩，之后才允许进入视觉包装、内容扩展和发布。

本报告之后，任何“继续美化首屏”“多加几个按钮”“多写几句随机日志”的方案都不应被视为重制进展。
