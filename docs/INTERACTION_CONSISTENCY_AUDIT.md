# CANOPY 交互一致性审计

日期：2026-07-14
范围：当前代码的模拟、生成、渲染、碰撞、聚焦、高亮与提示真值。此报告是代码审计；浏览器实机验证仍待补做。

## 结论

当前项目已经有统一 `affordance`、语义世界对象和单焦点高亮的正确基础，但仍存在“看起来一样、规则不同”与“系统声称可交互、实际永远做不到”的发布阻断问题。修复优先级不是继续增加资源种类，而是让每一种可见语义都满足：

`可辨认外观 → 唯一聚焦 → 清楚动作 → 可达条件 → 同一命令 → 可见结果`

## P0：破坏玩家对世界规则的信任

### P0-1 存在永久不可处理的交互对象

- 生成器会产生需要 tier 2 工具的铁木与大型岩体。
- 当前正常流程只提供 tier 1 斧和镐，既没有可达的升级配方，也没有替代加工路径。
- 固定 5×5 区块样本中，596 个交互对象有 61 个永久无法处理。
- 这不是“高级区域”的合理门槛：玩家看到高亮和工具需求，却无法在游戏内找到任何达成方式。

验收：

- 每个生成对象的 `minimumTier` 必须能从可达配方图中证明存在路径。
- 若暂不实现高级工具，则先让 tier 1 工具可处理全部树/岩石，并以工作时间、耐久损耗和产量表达差异。
- 增加全生成目录可达性测试：任一 `interactive=true` 对象都必须有至少一个可执行动作路径。

### P0-2 岩石语言混乱

- 可拾取散石、小型可采岩、不可聚焦 pebble clutter 与岩棚使用相近的多面体轮廓，尺寸区间也有重叠。
- 相似外观对应 `pickup`、`mine`、`none` 三套规则；只有准星扫到后才知道答案。
- flint/clay 的预览文案承诺不同材料，但当前实际产出仍统一落到 `stone`，进一步破坏辨识与收集期待。

验收：

- 散石必须有“可直接拾取”的小尺度、落地姿态和高亮语言。
- 矿体必须有明显体块、材质/色带与镐击反馈；所有矿体都可开采。
- 纯装饰碎石不得进入可拾取尺寸/轮廓区间；岩棚必须拥有明确入口、顶棚与箱体视线。
- flint/clay 要么交付对应物品与用途，要么在产出前移除误导性预览。

### P0-3 植物外观没有表达动作差异

- 四种语义植物目前都投影为同一种圆锥体。
- 它们分别要求手采、刀割或拾取，但画面没有叶型、茎蔓、果实或切割部位差异。

验收：

- 每个动作族至少有一个稳定轮廓：手采阔叶、刀割藤蔓、果实灌木。
- `resolveAffordance()`、聚焦高亮与模型上的交互点必须共享同一语义对象。
- 不允许只靠名字或提示文字区分高频资源。

### P0-4 水体可见范围与取水真值不一致

- 程序化地形会显示河段、湿地与水洼，但只有 authored stream 的小范围实体可以取水。
- 玩家站在视觉水面上仍可能得到“没有水源”的结果。

验收：

- 所有达到可取水深度的视觉水体都必须投影成统一的水源 affordance。
- 碰撞中的 `inWater`、画面水面、取水范围与地图水系必须来自同一世界水场。
- 不可取水的浅泥或积水必须在外观和提示上明确区分。

### P0-5 营火状态与模型状态不一致

- `fireLit=false` 时仍可能保留火焰模型。
- 添柴会改变数字燃料，却不改变木柴、火焰、光照、火星或声音。

验收：

- 熄灭时火焰与动态光必须关闭，只保留余烬/冷灰状态。
- 添柴成功后 1 秒内，木柴档位、火焰/光、短促火星、声音与燃料条同时变化。
- 满燃料拒绝不得消耗木棍或播放成功表现。

## P1：需要在下一批纵切统一

### 树木

- 所有语义树已经具备稳定 ID，但目前需完成站立、受损、倒伏、拾枝、分段、原木、树桩状态链。
- 不能再由 `quantity` 同时代表耐久、掉落和存在状态。
- 普通命中不直接生成木棍；材料必须存在于可见倒木上。
- 外观相同的语义树遵守相同规则；树种/年龄只改变时间、工具损耗和产出。

### 动物与危险

- 剧情蛇已迁移到 wildlife 协议；后续动物也必须使用同一生命、攻击、受伤、死亡、尸体与重生真值。
- 不允许再增加“走进半径直接结算”的隐藏危险。
- 普通程序化动物目前仍缺尸体待取协议；背包满时的掉落处理不能长期与剧情蛇分叉。

### 建筑与容器

- 建筑可交互锚点必须位于玩家实际能站到的位置，不能埋在碰撞体内部。
- 建筑模型、放置预览、碰撞和 `isNearCampStructure` 必须读取同一几何规格。
- 烟熏架已是正确范例：本地进度、环境阻塞、产出状态与模型共用结构实例 ID。

### 提示与输入

- HUD 不能对 blocked、avoid、movement 目标统一画成可按 `E`。
- 应由 affordance 统一输出 `execute | inspect | movement | unavailable`，键鼠、触控和后续手柄只负责映射，不重复判断。

## P2：画面升级时处理

- 建立连续世界场，让群系、地形高度、水系、地表材质和生态生成读取同一低频场，消除区块棋盘感。
- 为树、岩、植物建立少量高辨识轮廓族，而不是增加大量同轮廓换色。
- 用群落构图、林冠层次、前中远景雾与统一天气光照提高世界密度；不以单纯增加三角形作为目标。
- 低画质只能减少微型 clutter、粒子和阴影，不能隐藏任何可交互对象或其反馈。

## 自动化门槛

1. 生成可达性：扫描固定种子和 5×5 区块，所有交互对象都有可达动作路径。
2. 语义唯一性：每个焦点 ID 恰好映射一个模拟对象、一个 affordance 和一个交互锚点。
3. 外观族约束：`pickup/mine/cut/chop/collect-water` 不得共享完全相同的模型规格。
4. 跨层一致性：渲染可见、碰撞可达、命令距离与提示距离使用同一几何函数。
5. 状态反馈：资源数量、生命、燃料或进度变化后，下一渲染快照必须反映变化。
6. 存档往返：倒木、矿体、植物刷新、动物伤亡、建筑过程往返存档不重置也不重复发奖。
7. 低画质等价：标准/低画质的可交互 ID 集合完全一致。

## 实施顺序

1. 树木倒伏与原木纵切，同时消除 tier 2 不可达树。
2. 修复岩石/碎石外观语言与高级矿体可达性。
3. 统一程序化水体取水真值。
4. 植物轮廓与动作族升级。
5. 营火多通道反馈。
6. 浏览器恢复后走固定路线，验证准星、高亮、距离、夜间与触控手感。
