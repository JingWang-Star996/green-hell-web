# CANOPY：雨林密度、植物与立体路线实施方案

日期：2026-07-15
状态：执行合同；玩法层优先，Valheim 风格材质与光照重构仍在后置闸门

## 1. 核心游戏哲学

所有新增内容必须同时回答三个问题：

1. **生存趣味**：它是否制造了可观察、可计划、可反制的资源或风险？
2. **世界交互**：玩家能否从轮廓、位置和状态预测动词，而不是靠试按或记例外？
3. **热带雨林想象**：它是否表现出湿热、遮蔽、分层、竞争、腐败、再生和立体路线，而不是把绿色模型均匀撒在平地上？

热带雨林不是“树很多”的同义词。Smithsonian 将其概括为林冠、林下层、林地表等垂直层次，林冠会显著削弱下层的光与风；藤本则是有长而柔韧攀援茎、借助其他植物获得高度的独立结构群。[Smithsonian 雨林分层](https://striresearch.si.edu/rainforest/home/rainforest-layers/)，[Smithsonian 新热带藤本说明](https://naturalhistory.si.edu/research/botany/research/lianas-and-climbing-plants-neotropics/lianas-introduction)

因此，目标不是把可交互树数量暴力翻倍，而是用**语义近景 + 廉价中景 + 连续远景 + 雾与声音**共同制造体积，并保持“同形同规则”。

## 2. 当前问题的系统性成因

当前空旷感来自五个同时存在的缺口：

- 树冠高度、树干粗细和叶片尺度集中，缺少巨木—幼树—草本的比例对比。
- 离散树之间没有中景叶团、藤幕、倒木和地表覆盖，视线能无代价穿透整个区块。
- 地形虽有低频起伏，但没有形成“谷底—河岸—坡肩—高地”的路线决策，也没有陡坡的移动规则。
- hash 群系与实体河流没有共同地理真相，植物/石块出现位置不能解释环境。
- 植物大多只换颜色和名称，没有“看见特征—选择工具—采收—加工—用途—再生”的闭环。

## 3. 五层世界密度架构

| 层 | 距离 | 内容 | 交互合同 | 成本策略 |
|---|---:|---|---|---|
| L0 连续地形 | 全程 | 河谷、坡肩、台地、冲沟、泥地 | 行走、水、建造和遮挡共享同一高度/坡度真相 | 单一程序化高度场 |
| L1 语义近景 | 0–32m | 树、矿体、资源植物、倒木、动物 | 全部有稳定 ID、能力、碰撞、状态和差量存档 | 跨活动环实例池 |
| L2 近地被 | 0–24m | 蕨、落叶、草丛、小型阔叶 | 明确 `never-focus`，不冒充资源；可被风/脚步压动 | 每类一个 InstancedMesh |
| L3 中景遮蔽 | 20–70m | 大叶团、藤幕、幼树簇、树根板、冠下暗块 | 纯构图，不使用与可采集植物相同的完整轮廓 | 少量交叉叶片/低面簇、分块合批 |
| L4 远景冠层 | 55–180m | 冠层墙、突生巨木剪影、山脊植被带 | 不可聚焦、不参与命中 | 低频轮廓带、雾中 LOD/公告板 |

规则：

- L3/L4 可以“骗”出数量，但不能伪装成近处可采集对象；接近时必须淡出或由 L1/L2 的真实对象接替。
- 营地、河岸取水点、地标入口和主路线使用确定性净空区；不是均匀删树，而是形成空地—收窄—再开阔的视线节奏。
- 每区块密度由群系、坡度、离河距离、林隙扰动和性能档共同决定；不由纯随机常数决定。
- 移动端先削减 L4 段数和 L2 密度，再降低材质/阴影；不能删除会改变玩家判断的 L1 对象。

### 首个性能预算

- 活动 3×3 区块内，L1 树/岩/植物仍由语义生成唯一拥有。
- L2 每区块 24–48 个实例，按 3–4 种网格跨活动环合池。
- L3 每区块 12–24 个叶团/藤幕实例，最多 3 个材质桶。
- L4 每可见方向 6–12 个冠层轮廓段；不投射阴影，不进入射线检测。
- 新层在同一画质档的目标是增加不超过 8 个稳定 draw submissions；最终数字以 V0 浏览器基线为准。

## 4. 植物系统不是目录，而是循环

植物条目必须包含：群系/地形倾向、可读轮廓、采收动词、工具门槛、主要与副产物、加工方式、营养/医疗/建造用途、风险、再生阶段和动物关系。

首批路线：

| 植物原型 | 位置与轮廓 | 采收/加工 | 小—中循环价值 |
|---|---|---|---|
| 芭蕉/大蕉 | 林缘、河谷湿润林隙；巨型螺旋叶、假茎、垂果 | 徒手摘熟果；刀切叶/纤维；青果需烹饪 | 早期碳水；叶片用于包裹、棚顶与集水；果串吸引小动物 |
| 纤维藤 | 林缘、倒木与受扰林隙；跨树垂挂 | 地面断藤徒手拾，活藤需刀切 | 绑扎、工具、路线绳；过度采集降低局部藤本覆盖 |
| 棕榈果丛 | 棕榈林/湿地边缘；扇叶与高位果串 | 徒手低果，长柄/砍切高果 | 脂肪/碳水补给；与动物觅食竞争 |
| 巴西坚果巨木 | 成熟常绿林；高大板根、地面硬果 | 捡落果后敲开/烘烤 | 高脂长途口粮；低频丰产，不按普通草本刷新 |
| 螺旋花/蝎尾蕉原型 | 潮湿林下；高对比苞片 | 观察/少量采切 | 作为水热群系指示物、授粉动物热点；不把所有鲜艳植物都做食物 |
| 凤梨科蓄水株 | 树根/倒木/冠下；莲座叶杯 | 检查后取少量水 | 应急水源，但有污染/寄生虫概率，形成煮水判断 |
| 苦根薯原型 | 林缘/旧营地；掌状叶、土堆 | 挖根→切碎/浸泡/发酵→加热 | 高碳水中循环，但错误处理有中毒风险 |
| 药用阔叶/驱虫草 | 不同湿度与海拔的小簇 | 识别后切取、研磨/煮制 | 医疗补给；外观与普通地被严格区分 |

芭蕉优先不是装饰选择。Kew 记录香蕉是大型多年生草本而非树，叶片可达约 2.7m，并具有食物、纤维、包裹、遮蔽和建筑材料等多种用途；大蕉类通常更淀粉质、需要烹饪。这正好能把食物、建造和加工接成早期闭环。[Kew Musa acuminata 资料](https://powo.science.kew.org/taxon/urn%3Alsid%3Aipni.org%3Anames%3A797527-1/general-information)

螺旋花原型可承担“湿热环境指示物”而非普通掉落表。Kew 记录 `Heliconia rostrata` 原生于南美西部、以湿热带为主，并出现在森林/林地等环境。[Kew Heliconia rostrata](https://powo.science.kew.org/taxon/urn%3Alsid%3Aipni.org%3Anames%3A796978-1/general-information)

苦根薯只作为明确标注的**游戏化加工风险**，绝不把游戏配方当现实求生指南。FAO 资料指出木薯含氰苷，必须通过合适加工降低毒性；简单烹煮并非对所有品种都可靠。[FAO 木薯加工背景](https://www.fao.org/4/x5426e/x5426e03.htm)，[FAO 根茎与香蕉营养安全章节](https://www.fao.org/4/t0207e/T0207E08.htm)

## 5. 立体地形与路线系统

直接提高高度振幅会制造“镜头瞬移爬墙”和任务软锁，因此按以下依赖推进：

### VRT-1：共享地形字段

- 将高度拆为大陆尺度、群系尺度和局部侵蚀三层噪声。
- 河流切谷优先于 hash 群系；离河距离参与湿地、冲积石与植物分布。
- 营地/气象站只做小型局部平整，不抹平两地之间的坡谷。
- 输出 `height / slope / surface / traversalClass`，渲染、移动、建造、动物和资源位置共同消费。

### VRT-2：坡度移动真相

- `walkable`：正常移动。
- `scramble`：降低速度/增加体力，只允许短坡。
- `blocked`：不可直接攀升；必须绕路或使用路线设施。
- 下坡过陡会滑落/受伤，不能继续让相机无代价贴着高度场上升。

### VRT-3：稳定路线锚点

- 在可解释的崖边、沟谷、巨树根与河岸生成稳定 `TraversalAnchor`。
- 首批动词：放绳下降、藤索攀爬、倒木桥、简易梯、岩缝侧移。
- 锚点拥有两端位置、允许方向、体力成本、所需物品和遮挡体积；路线动画只是表现，模拟负责入口/出口结算。
- 作者路线保证主线可达，程序化锚点提供探索捷径；没有材料时永远存在绕路。

### VRT-4：玩家建设路线网络

- 可在合法锚点消耗藤条/绳索/木桩建立永久捷径。
- 已建路线成为地图标记和返程资产；雨季、腐坏与动物扰动形成维护需求。
- 不做任意表面攀爬；先用离散锚点保证浏览器端碰撞、动画和存档可靠。

## 6. 资源导演与生态边界

- 资源导演只在合法生态候选中调度到期节点，不能突破群系、容量、刷新窗口或在玩家眼前补货。
- 物理河岸优先于 hash biome：冲刷小石、断落藤条和湿生阔叶先看 `riverDistance`。
- 动物数量继续由承载力和种群账本决定；玩家缺食物不能凭空刷动物。
- 树木按树桩→幼苗→幼树→成熟树的多日阶段恢复，古树不进入快速循环。
- 玩家需求影响“哪个到期节点先恢复”，不改变单次产量；防止丢弃物资刷供给。

资源 Director 借鉴的是 Valve 所称的“设计师约束下的结构化不可预测性”、活动区与潜在可见性思想，不复制求生之路的敌人节奏。合法位置和生态规则由世界设计提供，导演只选择本轮激活哪一个。[Valve《The AI Systems of Left 4 Dead》](https://steamcdn-a.akamaihd.net/apps/valve/2009/ai_systems_of_l4d_mike_booth.pdf)，[Valve《Replayable Cooperative Game Design: Left 4 Dead》](https://cdn.fastly.steamstatic.com/apps/valve/2009/GDC2009_ReplayableCooperativeGameDesign_Left4Dead.pdf)

## 7. 实施顺序与闸门

1. **R1 可玩切片**：芭蕉映射现有棕榈果营养，加入可读大叶轮廓；L2/L3 密度层；河岸石/断藤保底；火把与移动 UI 回归。
2. **R2 资源导演**：合法到期节点的需求排序、离视野结算、保存 epoch、防刷测试。
3. **R3 world-v2 地形**：共享地形字段、真实坡度移动、作者路线；用户已允许研发期放弃旧存档，但仍需新生成版本与 fail-closed 存档命名空间。
4. **R4 路线设施**：绳索、倒木桥、梯子和地图捷径。
5. **R5 植物加工**：芭蕉叶/纤维副产物、苦根薯加工、蓄水植物、动物取食回写。
6. **V1 视觉表现**：在上述语义稳定后升级材质、光雾、风和后处理；不能用画面掩盖交互例外。

## 8. 硬验收

- 站在营地向任意方向看，30m 内至少出现前景遮挡、中景叶团和远景冠层三层，不再看穿整个区块。
- 390×844 与 844×390 下密度降级不删除真实资源，p95 帧时仍在移动端预算内。
- 芭蕉可仅凭轮廓识别；采果、食用和再生跨保存一致，装饰大叶不使用同一完整果株轮廓。
- 河岸 72m 路线内稳定存在徒手石与断藤，且节点不在准星前凭空刷新。
- 任意新可见离散树、岩或植物都能解释其基础动词；纯中远景对象靠 LOD 转换而不是突然消失。
- 地形升级后，主线至少有一条无设施可达路线和一条玩家可建设捷径；陡坡不能无代价直爬。
- 一百万 tick 后无 NaN、无超容量、无同点整批刷新，未改变区块不增加差量存档。
