# CANOPY：玩家状态、死亡复盘与检查点时间线

> 目标：玩家可以在死亡前读懂风险、在受伤时理解来源、在死亡后说清因果，并能回到一个由自己选择的可信恢复点。

## 1. 设计依据

Microsoft 的 Xbox Accessibility Guideline 103 要求会影响成败的关键信息不能只依赖单一感官，也不能只用颜色表达；其 Grounded 示例同时使用伤害方向光晕、生命变化和声音。[XAG 103](https://learn.microsoft.com/en-us/xbox/accessibility/xbox-accessibility-guidelines/103)

Microsoft 的存档建议强调原子更新、版本恢复和本地优先；其无障碍存档标准要求手动档与自动档分开、自动档不能覆盖手动档，并建议提供多个恢复位置。Grounded 还允许玩家调整自动保存间隔与保留数量，使玩家能从坏自动档之前恢复。[Game Saves](https://learn.microsoft.com/en-us/gaming/gdk/docs/features/common/game-save/game-saves)，[Accessibility Feature Tags](https://learn.microsoft.com/en-us/xbox/accessibility/accessibility-feature-tags)，[XAG 108](https://learn.microsoft.com/en-us/xbox/accessibility/xbox-accessibility-guidelines/108)

这些资料提供原则，不直接决定 CANOPY 的数值。下文是结合当前浏览器单机、Toy 云配额和雨林生存节奏作出的项目决策。

## 2. 失败定义

以下任意一种都算系统失败：

- 玩家看到生命下降，却不知道是蛇、捕食者、出血、感染、脱水还是工作快进造成。
- 状态只改变一根细条或一个颜色，移动端、色觉差异或强雨画面下看不见。
- 相同警告每个 tick 重复刷屏，最终让玩家忽略所有提示。
- 死亡界面只写“你死了”或“生命归零”。生命归零是结果，不是原因。
- 自动档覆盖手动档，或唯一自动档恰好保存于不可挽回的失败状态。
- 休息后死亡只能回到新游戏，而不能回到本次休息前。
- 存档菜单只显示内部时间戳，不能判断“我会回到哪里、当时状态如何”。

## 3. 多级状态反馈

### 3.1 四级强度

| 级别 | 含义 | 表现 | 交互要求 |
| --- | --- | --- | --- |
| `observe` | 趋势刚出现，尚未危险 | 手表/状态区出现有文字的状态签；首次出现一条短回执 | 不抢输入、不遮挡准星 |
| `warning` | 若继续当前行为会形成明显损失 | 图标形状 + 状态名称 + 一句后果；身体对应部位或来源方向提示 | 保持数秒，进入手表/身体页可查看处理 |
| `critical` | 短时间内可能失去行动能力或死亡 | 持续但克制的屏幕边缘节律、明确文字、独特音频事件；HUD 状态常驻 | 告诉玩家立刻可做的一个动作，不只写“危险” |
| `impact` | 刚刚发生伤害或状态跃迁 | 伤害值/强度、来源、方向（若权威数据存在）、命中身体部位、短暂冲击反馈 | 不能被制作/背包面板盖住；同一事件只结算一次 |

颜色只负责加强层级，不能承担含义。每个状态至少同时有：形状或图标、可读名称、严重度文字；游戏关键伤害还要有视觉之外的音频事件。当前版本可先复用代码合成提示音，不为此引入大音频包。

### 3.2 权威数据模型

UI 不允许从 CSS 动画或“生命条突然变短”反推原因。模拟层应产生有稳定 ID 的事实：

```ts
type StatusSeverity = "observe" | "warning" | "critical";

interface DamageIncident {
  id: string;
  tick: number;
  causeCode: string;
  sourceEntityId?: string;
  sourceLabel: string;
  amount: number;
  bodyPart?: string;
  directionDegrees?: number;
  conditionIds: string[];
  knownCountermeasureId?: string;
}

interface StatusSignal {
  id: string;
  category: "injury" | "illness" | "nutrition" | "hydration" | "exposure" | "threat";
  severity: StatusSeverity;
  label: string;
  consequence: string;
  actionLabel?: string;
  startedTick: number;
  updatedTick: number;
  sourceIncidentId?: string;
}
```

`causeCode` 是逻辑真值，中文文案是投影。蛇咬、撕裂伤跳血、失血、感染发热、饥饿、脱水和工作快进必须有不同的原因码，不能都落成 `health-loss`。

### 3.3 趋势、迟滞和降噪

- 阈值跨越才升级；状态在恢复阈值以上保持一段时间才降级，避免临界值闪烁。
- 同类低级提示有冷却；升级、来源变化或新身体部位不受旧冷却吞掉。
- 一次命令产生的直接伤害、继发状态和任务事件共享 transaction/incident 关系，HUD 展示主结果，详情保留完整链。
- 打开面板时，`critical` 与新 `impact` 位于全局反馈层；普通 `observe` 留在面板内，不堆满屏幕。
- 方向只在模拟/命中查询有可靠来源坐标时显示；环境状态不伪造红色方向箭头。

## 4. 死亡因果复盘

### 4.1 三层答案

死亡界面必须回答：

1. **直接死因**：最后把生命推到零的事实，例如“未处理的撕裂伤持续失血”。
2. **可干预链**：最多 3–5 个此前发生、且玩家当时可以改变的事实。
3. **已知改进方向**：只引用玩家已经发现的笔记/配方，不泄露未知答案。

示例：

```text
直接死因：持续失血
13:42 被捕食者击中左臂 → 形成撕裂伤
13:45 使用脏敷料 → 伤口仍在出血
14:03 超重奔跑 → 耐力耗尽，无法脱离第二次攻击
建议：你已记录“干净绷带可先止血”。
```

没有完整链时要诚实降级为“最近可信事件”，不能编造推理。死亡事件应冻结模拟，先完成本地恢复点索引读取，再开放按钮。

### 4.2 恢复入口

死亡页的主按钮是“选择恢复点”，而不是“重新开始”。并列入口：

- 推荐恢复点：最新且通过安全性检查的检查点。
- 查看全部：进入检查点时间线。
- 新的远征：明确提示会创建新 run，不覆盖手动档。

## 5. 检查点时间线

### 5.1 槽位类别

| 类别 | 默认数量 | 覆盖规则 |
| --- | ---: | --- |
| 手动存档 | 3 | 只由玩家明确覆盖/删除；自动存档永不写入 |
| 自动检查点 | 10 | 按时间轮转；设置中可选 4、10、20 |
| 导入前恢复点 | 1 | 每次成功导入前原子替换；不计入自动档 |
| 当前会话临时档 | 1 | IndexedDB 不可用时只在内存存在，并明确标为“本次会话” |

Toy 云优先同步手动档、当前活动档和最近的安全自动档；完整本地时间线仍受浏览器配额约束。云端 manifest 必须列出实际已同步槽位，不能把“本地有 10 个”误报成“云端都有 10 个”。

### 5.2 自动检查点触发

按玩家预期排序，事件可以合并，但原因必须保留：

- **休息前**：在时间推进、腐败、燃料消耗和状态结算之前提交；这是死亡回退的稳定锚点。
- 休息后：仅在玩家存活、结算完成且状态可操作时提交。
- 任务阶段完成、关键地标发现、关键建筑完成。
- 进入明显危险远征前的安全边界；不在伤害已经命中后伪装成“出发前”。
- 周期自动档、页面隐藏/退出；周期档不得比事件档更新却更不安全。

自动档禁止在以下状态成为唯一最新恢复点：已经死亡、命令结算中、导入未完成、世界 delta 写入一半、下一固定步确定必死。若无法判断“必死”，至少保留一个更早的安全档，不覆盖整个轮转窗口。

### 5.3 时间线卡片

每张恢复卡必须显示：

- 游戏日与时间、现实保存时间、游玩时长。
- 保存原因：休息前、任务完成、手动、远征边界等。
- 地点/群系与当前目标。
- 生命和最严重的两个状态；是否在暴雨、战斗或危险区。
- 本地/云端/本次会话徽标以及校验状态。

默认推荐“最新安全档”，但玩家可以选择更早的档。加载前展示预览与确认，加载失败不破坏当前存档。

## 6. 写入与恢复契约

1. 写临时 envelope，包含 schema、world version、runEpoch、simTick、checksum 和 checkpoint metadata。
2. 读回并校验临时 envelope。
3. 原子替换目标槽；更新 timeline manifest。
4. 本地成功后才异步排队 Toy 云写入。
5. 云失败不回滚本地成功；UI 显示“已保存在本地，云同步待重试”。
6. 加载只接受校验成功且版本兼容的 envelope；失败时尝试同槽 backup，再让玩家选择其他槽。

导入文件是非可信输入：限制大小、深度和节点数，校验格式、checksum、世界/内容版本和数值边界；成功导入前先保留 `preimport` 恢复点。

## 7. 分阶段实现

### Slice A：状态与直接伤害

- 权威 `DamageIncident` 与纯 `deriveStatusSignals()`。
- 蛇、捕食者、出血、感染、脱水、饥饿六条路径都有唯一原因。
- HUD 多级提示和面板上层全局 `impact`。

### Slice B：死亡复盘

- 有界因果事实缓存；死亡摘要选择器。
- 死亡界面显示直接原因、链和已知建议。

### Slice C：本地检查点时间线

- 3 手动 + 10 自动 + preimport；休息前/后触发。
- 死亡页选择与预览恢复点。

### Slice D：Toy 云与可配置保留

- 云 manifest/配额预算；冲突与部分同步可见。
- 自动档数量设置、导出/导入时间线包。

## 8. 硬验收

- 盲测中至少 90% 的死亡玩家能说出直接死因，至少 75% 能说出一个此前可改变的选择。
- 强雨、夜晚、1080p、手机竖屏和静音条件下，关键伤害仍可读；颜色关闭后含义不丢失。
- 休息开始前强制崩溃、休息结算中崩溃、休息后死亡三种情况都能回到本次休息前。
- 连续制造 25 个自动档后，手动档未被覆盖，时间线只保留设置数量，所有索引无悬挂项。
- 任意槽损坏只隔离该槽；当前档和其他恢复点仍可打开。
- 导入失败、云失败或 IndexedDB 不可用都不会显示虚假的“已安全保存”。
